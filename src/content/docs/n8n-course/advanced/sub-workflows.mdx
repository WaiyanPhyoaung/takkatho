## Sub-Workflow ဆိုတာ ဘာလဲ?

**Sub-Workflow** ဆိုတာ Main Workflow အကြီးတစ်ခုကနေ တခြား Workflow သေးသေးလေးတစ်ခုကို လှမ်းခေါ်သုံးတဲ့ Pattern ပါ။ Programming မှာဆိုရင် **Function** တစ်ခု ရေးထားသလိုပါပဲ — တစ်ကြိမ် ဖန်တီးထားပြီး နေရာများစွာမှာ ပြန်ခေါ်သုံး (Reuse) လို့ ရပါတယ်။

Sub-Workflow မသုံးရင် ဘာဖြစ်မလဲဆိုတော့ Email ပို့တဲ့ Logic, Slack ပို့တဲ့ Logic တွေကို Workflow အသစ်ဆောက်တိုင်း ထပ်ခါတလဲလဲ ရေးနေရပါမယ် (Code Duplicated ဖြစ်မယ်)။ Sub-Workflow သုံးလိုက်ရင်တော့ "Email ပို့တဲ့ Sub-Workflow" တစ်ခုတည်း ဆောက်ထားပြီး Order Workflow တွေကနေဖြစ်ဖြစ်၊ Invoice Workflow ကနေဖြစ်ဖြစ် အဲ့ဒီတစ်ခုတည်းကို လှမ်းခေါ်သုံးလို့ ရသွားပါပြီ (DRY Principle ပါ)။

## Sub-Workflow ဖန်တီးနည်း

### Step 1: Reusable Workflow အရင်ဆောက်ပါ

ဥပမာ - "Send Notification" ဆိုတဲ့ Sub-Workflow တစ်ခု ဆောက်ကြည့်ရအောင် -

<BranchFlow
  input="Execute Workflow Trigger (Input Data လက်ခံမယ့်နေရာ)"
  condition='Channel = "slack"?'
  truePath={["Slack: Post Message"]}
  falsePath={["Gmail: Send Email"]}
  trueLabel="YES ✅"
  falseLabel="NO ❌"
/>

Sub-Workflow ရဲ့ နောက်ဆုံး Node မှာ **"Set: Return Result"** Node လေး ထည့်ပေးပြီး ခေါ်တဲ့သူ (Caller) ဆီကို Result ပြန်ပို့ပေးဖို့ လိုပါတယ်။

**Execute Workflow Trigger Node** — Sub-Workflow တွေအတွက် Manual Trigger နေရာမှာ ဒီ Node ကို သုံးရပါတယ် -

```
Expected Input (Main Workflow ကနေ လှမ်းပို့ပေးမယ့် Data ပုံစံ):
{
  "channel": "slack",     // ဒါမှမဟုတ် "email"
  "message": "...",
  "recipient": "...",
  "subject": "..."        // Email အတွက်သာ
}
```

### Step 2: Main Workflow မှ ခေါ်သုံးပါ

Main Workflow ထဲမှာ **"Execute Workflow"** Node ကို ထည့်ပါ -

- Workflow: [Send Notification ဆိုတဲ့ Workflow ကို ရွေးပါ]
- Mode: Run once with all items
- Input Data:

| Field | Value |
|-------|-------|
| channel | slack |
| message | `{{ $json.orderSummary }}` |
| recipient | #orders |

## Common Sub-Workflow Patterns

### Pattern 1: Shared Utility 

နေရာများစွာမှာ ခဏခဏ သုံးရတဲ့ အလုပ်လေးတွေအတွက် Utility Sub-Workflows တွေ ဆောက်ထားလေ့ရှိပါတယ် -
- **"Format Myanmar Phone Number"** — ဖုန်းနံပါတ်တွေကို +959... ပုံစံ ပြောင်းပေးဖို့
- **"Validate Email Address"** — Email ပုံစံ မှန်မမှန် စစ်ဖို့
- **"Calculate Tax"** — အခွန် တွက်ချက်ဖို့
- **"Generate Unique ID"** — UUID တွေ၊ Ticket Number တွေ ထုတ်ပေးဖို့

**"Format Phone Number" Sub-Workflow ရဲ့ ဥပမာ -**

```javascript
// Code Node in Sub-Workflow
const phone = $input.first().json.phone;

// ဂဏန်းမဟုတ်တာတွေ အကုန်ဖျက်မယ်
let clean = phone.replace(/\D/g, '');

// မြန်မာ Country Code မပါရင် ထည့်မယ်
if (clean.startsWith('09')) {
  clean = '+959' + clean.slice(2);
} else if (!clean.startsWith('959')) {
  clean = '+959' + clean;
} else {
  clean = '+' + clean;
}

return [{ json: { formattedPhone: clean, original: phone } }];
```

## Execution Modes

Execute Workflow Node မှာ သုံးလို့ရတဲ့ Execution Mode (၂) မျိုး ရှိပါတယ် -

**Mode 1: "Run once with all items"**
- Sub-workflow ကို တစ်ကြိမ်ပဲ ခေါ်တယ်။ ပါလာတဲ့ Items အားလုံးကို တစ်ခေါက်တည်းနဲ့ ပို့ပေးလိုက်တယ်။
- Batch Processing လုပ်ချင်ရင် ဒါကို သုံးတာ ပိုကောင်းပါတယ်။

**Mode 2: "Run once for each item"**
- Item တစ်ခုရှိရင် တစ်ကြိမ် ခေါ်တယ်။ (ဥပမာ Item ၁၀ ခုဆိုရင် Sub-workflow ကို ၁၀ ခါ ပြန်ခေါ်မယ်)
- Item တစ်ခုချင်းစီကို သီးခြားစီ Process လုပ်ချင်ရင် သုံးပါတယ်။

## Workflow Organization — Best Practices

### Naming Convention (နာမည်ပေးစနစ်)

Workflow တွေကို နာမည်ပေးတဲ့အခါ စနစ်ကျဖို့ အရေးကြီးပါတယ် -

```
ကောင်းသော နာမည်ပေးခြင်း:
✅ "Core: Send Email Notification"
✅ "Integration: Shopify Get Orders"
✅ "Utility: Validate & Format Phone"
✅ "Daily: Morning Report"

ရှောင်ကြဉ်ရမည့် နာမည်များ:
❌ "Workflow 1"
❌ "test"
❌ "new workflow copy (2)"
```

### Folder Structure

n8n မှာ Workflow တွေကို ဒီလို အုပ်စုခွဲ (Organize) ထားသင့်ပါတယ် -
- **Core (Sub-workflows)** — Send Notification လိုမျိုး အဓိက အစိတ်အပိုင်းတွေ
- **Integrations** — Shopify: Sync Orders, Google Sheets: Update လိုမျိုး တခြား App တွေနဲ့ ချိတ်တဲ့အပိုင်းတွေ
- **Scheduled** — Daily Report ပို့တာမျိုး အချိန်နဲ့ Run တဲ့ Workflow တွေ
- **Triggers** — Webhook: Contact Form, Payment စသဖြင့် အပြင်ကနေ လာခေါ်မယ့် အပိုင်းတွေ

## Sub-Workflow vs Inline Logic

Workflow တစ်ခုတည်းမှာ အကုန်ဆက်ရေးမလား (Inline) သီးသန့်ခွဲထုတ်မလား (Sub-Workflow) ဆိုတာ နှိုင်းယှဉ်ချက်ပါ -

| | Sub-Workflow သုံးခြင်း | Inline အကုန်ရေးခြင်း |
|---|---|---|
| **Reuse လုပ်နိုင်မှု** | ✅ တခြားနေရာတွေကပါ လှမ်းခေါ်လို့ရတယ် | ❌ ဒီ Workflow တစ်ခုတည်းမှာပဲ သုံးလို့ရမယ် |
| **ပြင်ဆင်ရလွယ်ကူမှု** | ✅ တစ်နေရာတည်း ပြင်လိုက်ရင် အကုန်ပြောင်းသွားမယ် | ❌ နေရာတိုင်း လိုက်ပြင်နေရမယ် |
| **Testing** | ✅ သီးသန့်ခွဲပြီး စမ်းသပ်လို့ လွယ်တယ် | ❌ Workflow အကြီးကြီးထဲမှာ ရောနေလို့ စမ်းရခက်တယ် |
| **ရှုပ်ထွေးမှု** | ❌ အစမှာ Setup လုပ်ရတာ နည်းနည်းပိုရှုပ်တယ် | ✅ အလွယ်တကူ ဆက်ရေးသွားလို့ ရတယ် |
| **မြန်ဆန်မှု (Performance)** | ❌ Sub-Workflow ခေါ်ရတဲ့အတွက် နည်းနည်းပိုကြာနိုင်တယ် | ✅ တိုက်ရိုက်သွားလို့ ပိုမြန်တယ် |

<Aside type="tip" title="Sub-Workflow ကို ဘယ်အချိန် သုံးမလဲ">
ဒီ Logic ကြီးကို နောက်ထပ် **Workflow ၂ ခု ဒါမှမဟုတ် အဲ့ဒီထက်ပိုပြီး** ပြန်သုံးဖြစ်မယ်ဆိုရင်တော့ သေချာပေါက် Sub-Workflow သာ ဖန်တီးလိုက်ပါ။ တစ်ခါတည်းပဲ သုံးဖြစ်မယ်ဆိုရင်တော့ Inline ပဲ ရေးတာ အကောင်းဆုံးပါ။
</Aside>
