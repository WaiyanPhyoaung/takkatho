 Core Web Vitals တွေဟာ တစ်ခုနဲ့တစ်ခု အချင်းချင်း ဆက်စပ်နေတယ် ဆိုတာကို နားလည်ပြီး၊ ဒါတွေကို ဟန်ချက်ညီညီ ကောင်းမွန်အောင် လုပ်ဆောင်ဖို့အတွက် ဘယ်လို နည်းဗျူဟာ (Strategies) တွေ သုံးရမလဲဆိုတာ လေ့လာသင်ယူဖို့ ဖြစ်ပါတယ်။
</Aside>

## Vitals များ၏ အပြန်အလှန် ဆက်စပ်မှု

Core Web Vital တစ်ခုကို ကောင်းမွန်အောင် ကြိုးစား ပြင်ဆင်လိုက်တာဟာ၊ တစ်ခါတလေမှာ တခြား Vital တစ်ခုကို ပိုကောင်းသွားစေနိုင်သလို၊ မတော်တဆ ပိုဆိုးသွားစေတာမျိုးလည်း ဖြစ်နိုင်ပါတယ်။

- **ဥပမာ ၁:** Website ကို ပေါ့ပါးသွားအောင် လို့ JavaScript ဖိုင်တွေကို နောက်မှ တင်တာမျိုး (Defer လုပ်တာ) က INP (တုံ့ပြန်မှု) ကို ကောင်းသွားစေနိုင်ပါတယ်။ ဒါပေမဲ့၊ အကယ်၍ LCP Element (စာမျက်နှာရဲ့ အကြီးဆုံး ပုံ ဒါမှမဟုတ် စာ) ကို JavaScript သုံးပြီးမှ ဖန်တီးပြသထားတာမျိုး ဆိုရင်တော့ LCP ပေါ်လာဖို့ အချိန် ပိုကြာသွားစေမှာ ဖြစ်ပါတယ်။
- **ဥပမာ ၂:** ပုံတွေကို Lazy-loading (ပေါ်လာမယ့် အချိန်နီးမှ Load လုပ်ခြင်း) လုပ်တာက အစပိုင်းမှာ Data အသုံးပြုမှုကို လျှော့ချပေးပြီး LCP ကို ကောင်းစေပါတယ်။ ဒါပေမဲ့ ပုံတွေရဲ့ အရွယ်အစား (Width/Height Dimensions) တွေကို သေချာ ကြိုတင် မသတ်မှတ်ထားဘူးဆိုရင်၊ ပုံတွေ ပေါ်လာတဲ့အချိန်မှာ စာမျက်နှာကြီး ရွေ့သွားပြီး CLS (Layout Shift) ကို ပိုဆိုးသွားစေနိုင်ပါတယ်။

> ဒါကြောင့် ပြဿနာ တစ်ခုတည်းကိုပဲ ကွက်ပြီး မဖြေရှင်းဘဲ၊ သုံးစွဲသူရဲ့ အတွေ့အကြုံ (User Experience) တစ်ခုလုံးကို ထည့်သွင်းစဉ်းစားပြီး ပြည့်စုံမျှတတဲ့ နည်းလမ်း တစ်ခုကို အသုံးပြုဖို့ လိုအပ်ပါတယ် ခင်ဗျာ။

## ဦးစားပေး လုပ်ဆောင်ရမည့် နည်းဗျူဟာများ

- **အဆိုးဆုံး Vital ကို အရင် ပြင်ပါ။** Field Data (တကယ့် သုံးစွဲသူတွေဆီက ရတဲ့ အချက်အလက်) ကို အသုံးပြုပြီး သင့် Website ဟာ လက်တွေ့မှာ ဘယ် Core Web Vital မှာ အဆိုးဆုံး ဖြစ်နေလဲဆိုတာ အရင် ကြည့်ပါ။ အဲ့ဒီ အဆိုးဆုံး အပိုင်းကနေ စတင် ပြင်ဆင်ပါ။

- **အရေးကြီးတဲ့ Page တွေကို ဦးစားပေးပါ။** သုံးစွဲသူတွေ အဓိက အများဆုံး ကြည့်တဲ့ Page တွေ၊ စီးပွားရေးအရ အရေးပါတဲ့ Page တွေ (ဥပမာ - Homepage, Product Pages, ဈေးဝယ်ခြင်း ပြီးမြောက်တဲ့ Checkout စာမျက်နှာများ) ကို အရင်ဆုံး ကောင်းမွန်အောင် အာရုံစိုက် လုပ်ဆောင်ပါ။

- **လုပ်ရ လွယ်ကူတဲ့ အရာတွေကို အရင်လုပ်ပါ (Quick Wins)။** တချို့သော Optimizations တွေ ဥပမာ - ပုံတွေ (Images) မှာ Width/Height အရွယ်အစားတွေ ထည့်ပေးလိုက်တာမျိုးဟာ လုပ်ရတာ အလွန် လွယ်ကူပြီး၊ CLS အပေါ်မှာ ချက်ချင်းဆိုသလို ကြီးမားတဲ့ သက်ရောက်မှု (တိုးတက်မှု) ရှိစေပါတယ်။

- **တိုင်းတာပါ၊ ပြီးရင် ထပ်မံ မွမ်းမံ ပြင်ဆင်ပါ။** ကုဒ်တွေကို ပြောင်းလဲ ပြင်ဆင်မှုတွေ လုပ်ပါ။ ပြီးတာနဲ့ အဲ့ဒီ ပြင်ဆင်မှုတွေရဲ့ သက်ရောက်မှုကို ချက်ချင်း တိုင်းတာပါ (DevTools လို Lab Data ရော၊ တကယ့် Field Data မှာပါ စောင့်ကြည့်ပါ)။ လိုအပ်ရင် လိုအပ်သလို ထပ်မံ ပြင်ဆင်ပါ။

## အဖြစ်များတဲ့ အမှားများ (Common Pitfalls)

- **Metric တစ်ခုတည်းကိုသာ အလွန်အကျွံ အာရုံစိုက်ခြင်း:** LCP ကို ကောင်းအောင် အတင်းလုပ်ရင်းနဲ့၊ INP ဒါမှမဟုတ် CLS ကို ထိခိုက် ဆိုးရွားသွားအောင် မလုပ်မိပါစေနဲ့။ ဟန်ချက်ညီဖို့ အရေးကြီးပါတယ်။
- **Field Data ကို လျစ်လျူရှုခြင်း:** Lighthouse လို Lab Data တွေက Debug လုပ်ဖို့ အသုံးဝင်ပေမဲ့၊ တကယ့် သုံးစွဲသူတွေရဲ့ အတွေ့အကြုံ အစစ်အမှန်ကိုတော့ Field Data (CrUX Report) ကပဲ တိတိကျကျ ပြသနိုင်ပါတယ်။
- **Device နဲ့ Network မျိုးစုံမှာ မစမ်းသပ်ခြင်း:** Performance ဆိုတာ ကိုယ့်ရဲ့ မြန်ဆန်တဲ့ ကွန်ပျူတာ (သို့) Wi-Fi မှာ ကောင်းနေရုံနဲ့ မရပါဘူး။ မတူညီတဲ့ ဖုန်းအမျိုးအစားတွေ၊ နှေးကွေးတဲ့ 3G/4G Network တွေမှာ သိသိသာသာ ကွာခြားနိုင်ပါတယ်။
- **အချိန်ကြာလာတာနဲ့အမျှ Performance ဆိုးရွားသွားတာကို သတိမပြုမိခြင်း:** စဉ်ဆက်မပြတ် စောင့်ကြည့် (Monitor) မနေဘူးဆိုရင်၊ Features အသစ်တွေ ထပ်ထည့်လာတာနဲ့အမျှ Performance က တဖြည်းဖြည်းချင်း ကျဆင်းသွားနိုင်ပါတယ်။

<Aside title='💡 အနှစ်ချုပ် ပြန်လည် သုံးသပ်ခြင်း'>
- **Largest Contentful Paint (LCP)** က Website စတင် ပွင့်လာတဲ့ မြန်နှုန်း (အဓိက အကြီးဆုံး Content ပေါ်လာတဲ့ အချိန်) ကို တိုင်းတာပါတယ်။ **၂.၅ စက္ကန့် အောက်** ဖြစ်သင့်ပါတယ်။
- **Interaction to Next Paint (INP)** က သုံးစွဲသူ Website ထဲမှာ ရှိနေစဉ် တစ်လျှောက်လုံး အသုံးပြုမှု (Interactions) တွေအပေါ် Website ရဲ့ တုံ့ပြန်မှု မြန်နှုန်းကို ခြုံငုံ တိုင်းတာပါတယ်။ **200 ms အောက်** ဖြစ်သင့်ပါတယ်။ (ယခင်က သုံးခဲ့တဲ့ FID ဟာ ပထမဆုံး အသုံးပြုမှု တစ်ခုတည်းကိုသာ အဓိကထားခဲ့တာ ဖြစ်ပါတယ်)။
- **Cumulative Layout Shift (CLS)** က Website ရဲ့ ပုံရိပ် တည်ငြိမ်မှု (Content တွေ မမျှော်လင့်ဘဲ ဘယ်လောက် ရွေ့လျားသွားသလဲ) ကို တိုင်းတာပါတယ်။ **0.1 အောက်** ဖြစ်သင့်ပါတယ်။

ဒီ **Core Web Vitals** ၃ ခုဟာ အချင်းချင်း ခွဲခြားလို့မရအောင် ဆက်စပ်နေပါတယ်။ ဒါတွေကို ကောင်းမွန်အောင် လုပ်ဆောင်ဖို့အတွက် ပြီးပြည့်စုံတဲ့ နည်းဗျူဟာ တစ်ခု လိုအပ်ပြီး၊ အဓိက ရည်ရွယ်ချက်ကတော့ သုံးစွဲသူရဲ့ အတွေ့အကြုံ (User Experience) တစ်ခုလုံးကို ပိုမို ကောင်းမွန် ချောမွေ့လာအောင် လုပ်ဆောင်ဖို့ပဲ ဖြစ်ပါတယ်။ Chrome DevTools, PageSpeed Insights နဲ့ WebPageTest တို့လို Tools တွေဟာ ပြဿနာ ရှာဖွေ ဖြေရှင်းဖို့ (Debugging) အတွက် မရှိမဖြစ် အရေးကြီးပါတယ် ခင်ဗျာ။
</Aside>
