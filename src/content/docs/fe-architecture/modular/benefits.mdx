Modular Architecture ကို သုံးလိုက်ရင် ဘာတွေ ထူးခြားသွားမလဲဆိုတာ ကြည့်ရအောင်။

### ၁။ Scalability (တိုးချဲ့ရ လွယ်ကူခြင်း)
အိမ်တစ်လုံးမှာ အခန်းအသစ် ထပ်ဆောက်သလိုပါပဲ။ ကိုယ့် Website မှာ `Blog` ဆိုတဲ့ Feature အသစ် ထည့်ချင်တယ်ဆိုရင် `Blog` Module တစ်ခု သီးသန့်ဆောက်ပြီး ထည့်လိုက်ရုံပါပဲ။ ရှိပြီးသား `User` Module တို့ `Product` Module တို့ကို သွားပြီး ထိခိုက်စရာ၊ ပြင်စရာ မလိုပါဘူး။

### ၂။ Maintainability (ပြုပြင်ထိန်းသိမ်းရ လွယ်ကူခြင်း)
Bug တစ်ခု တက်ပြီ ဆိုပါစို့။ "Cart ထဲ ပစ္စည်းထည့်လို့ မရဘူး" ဆိုတဲ့ ပြဿနာပေါ်လာရင်၊ `Cart` Module ကိုပဲ တန်းသွားကြည့်လိုက်ရင် ပြီးပါပြီ။ Project ကြီးတစ်ခုလုံး အနှံ့ လိုက်ရှာနေစရာ မလိုတော့ပါဘူး။

### ၃။ Team Collaboration (ပူးပေါင်းဆောင်ရွက်မှု)
Team အရွယ်အစား ကြီးလာတဲ့အခါ လူတွေ အများကြီးနဲ့ အလုပ်လုပ်ရပါတယ်။
- Team A က `Checkout` Module ကို တာဝန်ယူမယ်။
- Team B က `Profile` Module ကို တာဝန်ယူမယ်။
ဘယ်သူက ဘယ်နယ်နိမိတ်ကို တာဝန်ယူရမလဲ ရှင်းလင်းသွားတဲ့အတွက် Code တွေ ငြိတာ၊ တစ်ယောက်ရေးတာ တစ်ယောက် သွားဖျက်မိတာ (Merge Conflict) တွေ အရမ်း နည်းသွားပါတယ်။

### လက်တွေ့ Project တွင် အသုံးပြုပုံ (Folder Structure)
Project အကြီးကြီးတွေမှာ Component အားလုံးကို `components` ဆိုတဲ့ Folder တစ်ခုတည်းမှာ အကုန်ပြုံထည့်လေ့ မရှိပါဘူး။ **Feature အလိုက် (Feature-based)** ခွဲပြီး ဖွဲ့စည်းကြပါတယ်။

```text
src/
  features/
    auth/          (Auth နဲ့ဆိုင်တဲ့ Module)
      components/
      hooks/
      services/
    product/       (Product နဲ့ဆိုင်တဲ့ Module)
      components/
      hooks/
      services/
    cart/          (Cart နဲ့ဆိုင်တဲ့ Module)
      components/
      hooks/
      services/
```
ဒီလို သေသေချာချာ ခွဲထားလိုက်ရင် `Auth` နဲ့ ပတ်သက်တာ မှန်သမျှ `auth` Folder အောက်မှာပဲ ရှိနေမှာဖြစ်လို့၊ ရှာရဖွေရ လွယ်ကူသွားပါတယ်။

<Aside title="SPA တိုင်း Modular ဖြစ်လား?" type="tip">
**ဟုတ်ပါတယ်။** React, Vue, Angular တို့လို Framework အားလုံးဟာ Component-Based ဖြစ်တဲ့အတွက် Modular သဘောတရား အလိုလို ပါဝင်ပြီးသားပါ။ ဒါပေမဲ့ Project ကြီးလာတဲ့အခါမှာတော့ Component တွေချည်းပဲ သက်သက် မဟုတ်တော့ဘဲ Folder Structure ကိုပါ Modular ဖြစ်အောင် (Feature-based ပုံစံမျိုး) ပြောင်းလဲ ဖွဲ့စည်းဖို့ လိုအပ်လာပါတယ်။
</Aside>
