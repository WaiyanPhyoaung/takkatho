<Aside title="မိတ်ဆက်">
  Frontend လောကမှာ Monolith ဆိုတာကို "ခေတ်ဟောင်း" ကြီးလို့ မထင်လိုက်ပါနဲ့။ အခုခေတ် React, Vue, Angular တို့နဲ့ ရေးထားတဲ့ Web Application ၉၀% လောက်က Monolith ပုံစံတွေချည်းပါပဲ။ ဒါကို နားလည်လွယ်အောင် **"Swiss Army Knife (ဓားစုံ)"** လေးနဲ့ နှိုင်းယှဉ်ကြည့်ရအောင်။
</Aside>

### Monolith ဆိုတာ တကယ်တမ်း ဘာလဲ?
Frontend Monolithic Architecture ဆိုတာ အလွယ်ပြောရရင် **"Single Deployment Unit" (တစ်ခုလုံးကို တပေါင်းတည်း တင်ရတဲ့ စနစ်)** ပါ။

<ContentImage src={armyKnife} alt='swiss army knife' width='400' />

Swiss Army Knife (ဓားစုံ) တစ်ချောင်းကို မြင်ယောင်ကြည့်ပါ။
- ဓားပါတယ်၊ ကတ်ကြေးပါတယ်၊ ဝက်အူလှည့်ပါတယ်၊ ဖောက်တံပါတယ်။
- ဒါပေမဲ့ အဲ့ဒီ ကိရိယာတွေ အားလုံးက **ကိုယ်ထည်တစ်ခုတည်း (One Body)** ထဲမှာ တွဲလျက်သား ဖြစ်နေပါတယ်။
- ဝယ်ရင်လည်း တစ်ချောင်းလုံး ဝယ်ရတယ်၊ ပျောက်ရင်လည်း တစ်ချောင်းလုံး ပျောက်ပါတယ်။ တစ်ခုချင်းစီ ဆွဲဖြုတ်ပြီး သုံးလို့ မရပါဘူး။

Web Development မှာလည်း ဒီသဘောပါပဲ။
- `Header`, `Footer`, `ProductPage`, `CheckoutPage` ဆိုပြီး အပိုင်းတွေ (Components) အများကြီး ရှိပါမယ်။
- ဒါပေမဲ့ Build လုပ်လိုက်ရင် `bundle.js` ဆိုတဲ့ ဖိုင်တစ်ဖိုင်တည်း (သို့မဟုတ် Folder တစ်ခုတည်း) ထွက်လာပါတယ်။
- ဆာဗာ (Server) ပေါ်ကို တင်ရင် (Deploy လုပ်ရင်) အပိုင်းလေးတစ်ခုကိုပဲ ရွေးတင်လို့ မရဘဲ အကုန်လုံးကို တပြိုင်တည်း တင်ရပါတယ်။ **"တခါပြင် တခါတင် (All-or-Nothing Deployment)"** ဖြစ်နေရင် အဲ့ဒါဟာ Monolith ပါပဲ။

### Monolith နဲ့ Spaghetti Code အထင်အမြင်လွဲမှားခြင်း

<ContentImage src={spaghetti} alt='spaghetti' width='400' />

လူတော်တော်များများက Monolith ဆိုတာနဲ့ "ရှုပ်ပွနေတဲ့ Code တွေ (Spaghetti Code)" လို့ မှားမြင်တတ်ကြပါတယ်။ တကယ်တော့ မတူပါဘူး။
- **Monolith:** ဓားစုံတစ်ချောင်း (အကုန်ပေါင်းပြီး တင်ရတဲ့ Deployment ပုံစံ) ဖြစ်ပါတယ်။
- **Spaghetti Code:** သံချေးတက်ပြီး ဖွင့်ရခက်နေတဲ့ ဓားစုံ (Code တွေ ရှုပ်ပွပြီး အရည်အသွေး ညံ့ဖျင်းနေခြင်း) ဖြစ်ပါတယ်။

ကောင်းမွန်တဲ့ Monolith (Modular Monolith လို့လည်း ခေါ်တယ်) ဆိုတာ ဆီထည့်ထားတဲ့ ဓားစုံလိုပါပဲ။ ကတ်ကြေးလိုရင် ကတ်ကြေးပဲ အလွယ်တကူ ဆွဲထုတ်လို့ရတယ်၊ ကျန်တာတွေ ပါမလာဘူး။ Code တွေကို Folder ခွဲပြီး စနစ်တကျ ရေးထားရင် Monolith က အရမ်းကောင်းပါတယ်။

### ဘာကြောင့် Monolith ကို ရွေးချယ်သင့်သလဲ?
အခုခေတ်မှာ Micro-Frontend တွေ ခေတ်စားနေပေမယ့် လက်တွေ့မှာတော့ Project အများစုအတွက် Monolith က အကောင်းဆုံး ရွေးချယ်မှု ဖြစ်နေဆဲပါ။ ဘာကြောင့်လဲဆိုတော့:

1.  **Simple to Develop (တည်ဆောက်ရ လွယ်ကူခြင်း):** ဓားတစ်ချောင်းတည်း ကိုင်သွားရုံပါပဲ။ (Code အားလုံးက နေရာတစ်နေရာတည်းမှာ (Single Repo) ရှိနေတဲ့အတွက် ရှာရလွယ်တယ်၊ ပြင်ရလွယ်ပါတယ်)။
2.  **Simple to Deploy (လွှင့်တင်ရ လွယ်ကူခြင်း):** အိတ်ကပ်ထဲ ထည့်သွားလိုက်ရုံပါပဲ။ (Build command တစ်ကြောင်းရိုက်၊ ထွက်လာတဲ့ဖိုင်ကို Server ပေါ်တင်လိုက်ရုံနဲ့ အလုပ်ပြီးသွားပါပြီ)။
3.  **Performance (စွမ်းဆောင်ရည်):** ဓားစုံဆိုတော့ လိုအပ်တဲ့အချိန် တစ်ခါတည်း ထုတ်သုံးလို့ ရပါတယ်။ (Browser က Bundle တစ်ခုတည်းကိုပဲ Load လုပ်ရလို့ Caching လုပ်ရတာ ပိုထိရောက်ပါတယ်)။

<Aside title="နိဂုံးချုပ်" type='tip'>
  Frontend Monolith ဆိုတာ "Single Page Application (SPA)" အများစုရဲ့ သဘာဝပါပဲ။ အထဲမှာ Components တွေ ဘယ်လောက်ပဲ အများကြီး ခွဲရေးထားပါစေ နောက်ဆုံးမှာ User ဆီကို **Unit တစ်ခုတည်း** အနေနဲ့ ရောက်သွားတာမျိုးပါ။ စီမံခန့်ခွဲရ အရမ်းလွယ်ကူပြီး၊ Project တစ်ခုစတင်ဖို့အတွက် အကောင်းဆုံး Architecture တစ်ခု ဖြစ်ပါတယ်။
</Aside>
