<Aside title="ရည်ရွယ်ချက်">
ဒီသင်ခန်းစာမှာတော့ အခုထိ လေ့လာခဲ့တဲ့ အဓိက Architectural Pattern တွေကို စုစည်းပြီး ပြန်လည်သုံးသပ်သွားမှာပါ။ ဒါ့အပြင် Frontend Development နယ်ပယ်ရဲ့ **အနာဂတ်ကို ဦးတည်နေတဲ့ နည်းပညာများ** (ဥပမာ- Server Components) ကိုပါ လေ့လာသွားမှာဖြစ်ပါတယ်။

Frontend Architect တစ်ယောက်အနေနဲ့ Application ရဲ့ လိုအပ်ချက်ပေါ်မူတည်ပြီး ဘယ် Pattern ကို ဘယ်လိုပေါင်းစပ်ဆုံးဖြတ်ရမယ်ဆိုတဲ့ အမြင်တစ်ခု ရရှိသွားမှာဖြစ်ပါတယ်။
</Aside>

### ဘာတွေလေ့လာရမလဲ
- သင်ခန်းစာအားလုံးမှ အဓိက Architectural ဆုံးဖြတ်ချက်များကို ပြန်လည်သုံးသပ်ခြင်း။
- Server Components ဆိုတာ ဘာလဲ၊ ဘာကြောင့် ဒါက Frontend ရဲ့ အနာဂတ်ဖြစ်လာနိုင်သလဲ။
- Web Components တွေနဲ့ Micro-Frontend တွေရဲ့ ဆက်စပ်မှု။
- နောင်အနာဂတ်မှာ Frontend Application တွေကို ပိုမြန်အောင်လုပ်မယ့် နည်းလမ်းများ။
- သင်ခန်းစာမှ ရရှိသော အသိပညာများကို လက်တွေ့ Project များတွင် အသုံးချပုံ။

### အသေးစိတ်ရှင်းလင်းချက်
### အဓိက Architectural ဆုံးဖြတ်ချက်များ ပြန်လည်သုံးသပ်ခြင်း
Frontend Architect တစ်ယောက်ရဲ့ အလုပ်က နည်းပညာတွေကို တစ်ခုချင်းစီသိရုံတင်မကဘဲ သူတို့ရဲ့ **အားသာချက်/အားနည်းချက်** တွေကို သိပြီး လိုအပ်တဲ့နေရာမှာ မှန်ကန်စွာ ပေါင်းစပ် အသုံးချတတ်ဖို့ပါပဲ။

| ဆုံးဖြတ်ရမည့် အပိုင်း | ရွေးချယ်စရာများ | ဘယ်အချိန်မှာ ရွေးမလဲ |
|--------------------|------------------|-------------------------|
| **Code Structure**     | Modular / Micro-Frontend | သင့် App ကို Team မတူတာတွေက သီးခြားစီ Deploy လုပ်ဖို့ လိုသလား - **(Micro-Frontend)** ဒါမှမဟုတ် Codebase တစ်ခုတည်းမှာပဲ စနစ်တကျ ခွဲခြားဖို့ လိုသလား - **(Modular)** |
| **Data Management**    | Flux/Redux / Simple state tools | သင့် App မှာ Global State ကို အလွန်အမင်း ထိန်းချုပ်ဖို့ လိုအပ်လား - **(Redux)** ဒါမှမဟုတ် Local State ကိုပဲ အဓိကထားမလား - **(Simple Tools)**|
| **Speed & Delivery**   | CSR / SSR / SSG / ISR | သင့် App မှာ SEO အလွန်အမင်း လိုအပ်လား - **(SSR/SSG)** ဒါမှမဟုတ် Login ဝင်ပြီးမှ သုံးရတဲ့ Dashboard လား - **(CSR)** |

Architectural ဆုံးဖြတ်ချက်တွေက တစ်ခုတည်းကို ရွေးချယ်ရတာမျိုးမဟုတ်ဘဲ **Hybrid (ရောနှော)** ပုံစံမျိုးနဲ့ အကောင်းဆုံး အဖြေကို ရှာဖွေရတာ ဖြစ်ပါတယ်။

### ရှေ့ဆက်လာမယ့် နည်းပညာ ခေတ်ရေစီးကြောင်းများ
Frontend နယ်ပယ်ဟာ အမြဲတမ်းပြောင်းလဲနေပါတယ်။ အခုလက်ရှိမှာ အပြောင်းအလဲအကြီးဆုံး ဖြစ်လာမယ့် နည်းပညာ နှစ်ခုကတော့-

    #### Server Components များ (Rendering ကို Server မှာပဲ လုပ်ခြင်း)
    အရင်က **CSR** မှာ Browser က Data ကို တောင်းပြီး render လုပ်တယ်။ **SSR** မှာ Server က HTML ကို ဖွဲ့ပြီး ပို့ပေးတယ်။ **Server Components** ကတော့ အဲ့ဒီနှစ်ခုကို ပေါင်းစပ်ထားပါတယ်။
        - **အဓိကအယူအဆ** - Website ရဲ့ Data Fetching နဲ့ Initial Rendering လုပ်ငန်းတွေကို **Server ဘက်မှာပဲ ပြီးအောင် လုပ်လိုက်ပြီး**၊ User နဲ့ တုံ့ပြန်ဆက်ဆံရမယ့် အပိုင်း (Click လုပ်တာ၊ Form ဖြည့်တာ) ကိုသာ **Browser** ဘက်ကို ပို့ပေးတဲ့ နည်းလမ်း ဖြစ်ပါတယ်။
        - **အားသာချက်** - Browser မှာ လုပ်ရမယ့် အလုပ်တွေ အရမ်းနည်းသွားတဲ့အတွက် Loading Speed အလွန်မြန်လာပါတယ်။ Server နဲ့ Database ကပ်လျက်ရှိတဲ့အတွက် Data ယူရတာလည်း ပိုမြန်ဆန်ပါတယ်။
        
        - **အသုံးပြုမှု** - Next.js နဲ့ React တို့က ဒီပုံစံကို ဦးဆောင်နေပါတယ်။


    #### Web Components နှင့် Framework များ မတူညီမှု လျှော့ချခြင်း
    Web Components ဆိုတာ Browser တွေက Native အတိုင်း ထောက်ပံ့ပေးတဲ့ UI Component တွေကို ဖန်တီးတဲ့ နည်းလမ်းတစ်ခုပါ။
        - **အဓိကအယူအဆ** - Micro-Frontend Architecture မှာ Component တွေကို React/Vue လို Framework ပေါ်မှာ မမှီခိုဘဲ Web Components နဲ့ တည်ဆောက်ထားရင်၊ ဘယ် Framework နဲ့ရေးထားတဲ့ Application မှာမဆို အလွယ်တကူ ပေါင်းစပ်သုံးစွဲနိုင်ပါတယ်။
        - **အားသာချက်** - ဒါက Micro-Frontend လိုမျိုး ကြိုက်နှစ်သက်ရာ framework သုံးပြီး ရေးတဲ့ နေရာတွေမှာ ပြန်လည်အသုံးပြုနိုင်ပြီး အများကြီး အထောက်အကူပြုနိုင်ပါတယ်။
        
    
### နိဂုံးချုပ်
Frontend Architecture ကို လေ့လာတယ်ဆိုတာက "အကောင်းဆုံး Pattern တစ်ခုတည်း" ကို ရှာတာမဟုတ်ပါဘူး။ Application ရဲ့ အရွယ်အစား၊ Team ဖွဲ့စည်းပုံ၊ Performance လိုအပ်ချက်နဲ့ Budget တို့လို **အခြေအနေမျိုးစုံကို ထည့်သွင်းစဉ်းစားပြီး** လိုအပ်တဲ့အချိန်မှာ လိုအပ်တဲ့ Pattern တွေကို **ရောနှော** အသုံးပြုတတ်ဖို့ပါပဲ။

ဒီသင်ခန်းစာတွေကနေ ရရှိတဲ့ အသိပညာတွေက သင့်ကို ခိုင်မာတဲ့ ဆုံးဖြတ်ချက်တွေ ချနိုင်တဲ့ Senior Frontend Devloper/Frontend Architect တစ်ယောက်ဖြစ်လာစေဖို့ အထောက်အကူပြုပါလိမ့်မယ်။

<Aside title="အဓိကမှတ်သားစရာများ" type="tip">
- Architectural ဆုံးဖြတ်ချက်တွေက **Project ရဲ့ လိုအပ်ချက်ပေါ်မှာ အမြဲမူတည်ပါတယ်။** (ဥပမာ- SEO လိုအပ်ရင် SSR/ISR ကို သုံးပါ။)
- **Modular** နဲ့ **Micro-Frontend** ဟာ Code တွေစုဖွဲ့ဖို့ မရှိမဖြစ် Pattern တွေ ဖြစ်ပါတယ်။
- Frontend ရဲ့ အနာဂတ်ဟာ **Server Components** တွေဆီ ဦးတည်နေပြီး Rendering နဲ့ Data Fetching တွေဟာ Server ဘက်ကို ပိုရောက်လာပါမယ်။
- **Web Components** တွေကတော့ နည်းပညာအမျိုးမျိုးကြား ချောမွေ့စွာ ပေါင်းစပ်နိုင်ပါမယ်။
</Aside>
