## MVP (Model-View-Presenter)

MVP Pattern မှာတော့ View ဟာ User Interface (UI) နဲ့ User တွေရဲ့ Actions တွေကိုပဲ တာဝန်ယူပါတယ်။ UI မှာ ဘာတွေပြရမလဲ၊ User က ဘာလုပ်လိုက်လဲဆိုတာကို Presenter ကို အကြောင်းကြားပေးပါတယ်။

Presenter ကတော့ View ကနေရလာတဲ့ User Actions တွေပေါ်မူတည်ပြီး Model ကိုခိုင်းစေကာ Data တွေတောင်းပါတယ်။ ရလာတဲ့ Data တွေကို UI မှာပြသနိုင်အောင် ပြင်ဆင်ပြီး View ကိုပြန်ပေးပါတယ်။ View နဲ့ Model ကြားမှာကြားခံအနေနဲ့ရှိနေပြီး Business Logic တွေကိုကိုင်တွယ်ဖြေရှင်းပေးပါတယ်။

Model ကတော့ Application ရဲ့ Data နဲ့ Business Logic အပိုင်းကိုကိုင်တွယ်ပါတယ်။ Data တွေကို သိမ်းဆည်းခြင်း၊ ပြင်ဆင်ခြင်း၊ ဖတ်ခြင်းစတဲ့အလုပ်တွေကိုလုပ်ဆောင်ပါတယ်။

<Aside type="note">
MVP မှာ View နဲ့ Presenter ဟာ တစ်ခုနဲ့တစ်ခု တိုက်ရိုက်ချိတ်ဆက်ထားပြီး၊ Presenter က View ကို Update လုပ်ပေးပါတယ်။
</Aside>

## နှိုင်းယှဉ်ချက်

| Feature | MVP (Model-View-Presenter) | MVVM (Model-View-ViewModel) |
| :--- | :--- | :--- |
| **View** | Passive View။ User Actions တွေကို Presenter ကိုလက်ဆင့်ကမ်းပေးတယ်။ | Active View။ Data Binding ကနေတစ်ဆင့် ViewModel နဲ့ချိတ်ဆက်ထားတယ်။ |
| **ချိတ်ဆက်ပုံ** | View နဲ့ Presenter က Interface ကနေတစ်ဆင့် တိုက်ရိုက်ချိတ်ဆက်ထားတယ်။ | View နဲ့ ViewModel က Data Binding နဲ့ Commands တွေကနေတစ်ဆင့် ချိတ်ဆက်ထားတယ်။ |
| **Testability** | Presenter ကို Test လုပ်ရတာလွယ်ကူတယ်။ View ကို Mock လုပ်ဖို့လိုတယ်။ | ViewModel ကို Test လုပ်ရတာလွယ်ကူတယ်။ View ကို Test လုပ်စရာမလိုဘူး။ |
| **Coupling** | View နဲ့ Presenter ကြားမှာ Coupling ရှိတယ်။ | View နဲ့ ViewModel ကြားမှာ Decoupled ဖြစ်တယ်။ |

## ဘယ် Pattern ကို ဘယ်အချိန်သုံးမလဲ

- **MVP** ကို View နဲ့ Business Logic ကို တိတိကျကျခွဲခြားချင်တဲ့အချိန်၊ Testability ကိုအလေးထားတဲ့အချိန်တွေမှာ သုံးနိုင်ပါတယ်။ View ကိုအလွယ်တကူ Mock လုပ်နိုင်တဲ့အတွက် Unit Test တွေရေးရတာလွယ်ကူစေပါတယ်။

- **MVVM** ကိုတော့ Data Binding ပါတဲ့ Framework တွေ (ဥပမာ - WPF, Angular, Vue, React) နဲ့အလုပ်လုပ်တဲ့အခါ၊ View နဲ့ Business Logic ကို Decouple လုပ်ချင်တဲ့အခါတွေမှာ သုံးနိုင်ပါတယ်။ Code တွေကို ပိုပြီးရှင်းလင်းအောင်၊ Maintain လုပ်ရလွယ်အောင် ကူညီပေးပါတယ်။
