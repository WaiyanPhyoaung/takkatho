### Component-Based Architecture ရဲ့ အားသာချက်များ
- **ပြန်လည်အသုံးပြုနိုင်ခြင်း (Reusability):** Component တစ်ခုကို မတူညီတဲ့ စာမျက်နှာတွေမှာ အကြိမ်ကြိမ် ပြန်သုံးနိုင်တယ်။

- **ပြင်ဆင်ထိန်းသိမ်းရလွယ်ကူခြင်း (Maintainability):** Component တစ်ခုမှာ bug ဖြစ်ရင် အဲ့ဒီတစ်ခုကိုပဲ ပြင်လိုက်တာနဲ့ သူ့ကိုသုံးထားတဲ့ နေရာတိုင်းမှာ အလိုလို ပြင်ပြီးသားဖြစ်သွားတယ်။

- **တိုးချဲ့ရလွယ်ကူခြင်း (Scalability):** Component အသစ်တွေ ထပ်ထည့်ပြီး app ကို အလွယ်တကူ တိုးချဲ့နိုင်တယ်။

- **အဖွဲ့လိုက်လုပ်ဆောင်ရလွယ်ကူခြင်း (Team Collaboration):** Developer တွေက မတူညီတဲ့ Component တွေကို တစ်ချိန်တည်းမှာ သီးခြားစီ လုပ်ဆောင်နိုင်တယ်။

### အားနည်းချက်များ နှင့် ထည့်သွင်းစဉ်းစားရန်များ
- Component တွေ သေးလွန်း၊ များလွန်းသွားရင် project က ပိုရှုပ်ထွေးသွားနိုင်တယ်။

- Application ကြီးတွေမှာ Component တစ်ခုနဲ့တစ်ခုကြား data ပေးပို့တာ (State/Props Management) ရှုပ်ထွေးနိုင်တယ်။

- Component တွေ အားလုံးက မျှသုံးရမယ့် data တွေအတွက် React Context, Redux, Vuex တို့လို state management နည်းလမ်းတွေ လိုအပ်လာတယ်။

### ဥပမာ
E-commerce ဝဘ်ဆိုဒ်တစ်ခုကို ဥပမာအနေနဲ့ ကြည့်ကြရအောင်။

- **ProductCard Component:** ကုန်ပစ္စည်းရဲ့ ပုံ၊ နာမည်၊ ဈေးနှုန်းကို ပြပေးတဲ့ အပိုင်း။

- **Navbar Component:** Logo နဲ့ menu တွေကို ပြပေးတဲ့ အပိုင်း။

- **Footer Component:** ဆက်သွယ်ရန်လိပ်စာနဲ့ link တွေကို ပြပေးတဲ့ အပိုင်း။

အပေါ်က **ProductCard Component** တစ်ခုတည်းကိုပဲ homepage, category page, search results page တွေမှာ ပြန်မရေးဘဲ အကြိမ်ကြိမ် ပြန်သုံးလို့ရပါတယ်။

<Aside title="အနှစ်ချုပ်" type='tip'>
- Component-based architecture ဆိုတာ UI ကို ပြန်သုံးလို့ရတဲ့ သီးခြားအစိတ်အပိုင်း (component) လေးတွေအဖြစ် ခွဲထုတ်တာ ဖြစ်တယ်။

- ဒါက app ကို ပြင်ဆင်ထိန်းသိမ်းဖို့၊ ပြန်သုံးဖို့၊ တိုးချဲ့ဖို့နဲ့ အဖွဲ့လိုက်လုပ်ဆောင်ဖို့ လွယ်ကူစေတယ်။

- Folder တွေကို စနစ်တကျဖွဲ့စည်းထားခြင်းက component တွေ အများကြီးကို ထိန်းချုပ်ရလွယ်ကူစေတယ်။

- Component တွေ သေးလွန်းများလွန်းတာ၊ state management နဲ့ props ပေးပို့တာတွေကိုတော့ သတိပြုရမယ်။
</Aside>
