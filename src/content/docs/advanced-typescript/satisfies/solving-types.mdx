## Type Annotation (:) ကြောင့် ဖြစ်တဲ့ ပြဿနာ - Type ကျယ်ပြန့်သွားခြင်း (The Widening Effect)

ပြီးခဲ့တဲ့ သင်ခန်းစာက `myAppConfig` ဟာ `AppConfig` ဖြစ်ကြောင်း Type Annotation ကို သုံးပြီး သေချာအောင် လုပ်ချင်ရင် ဘာဖြစ်မလဲ။

<Code lang="typescript" code={typeAnnotation} title="config.ts" />

ဒီလို `annotatedConfig: AppConfig` လို့ ရေးလိုက်တာနဲ့ TypeScript က `annotatedConfig` အကြောင်း ဘာတွေသိသလဲ ကြည့်ရအောင်။

- `annotatedConfig.env`: သူ့ရဲ့ Type က အခု `"development" | "production" | "test"` ဖြစ်သွားပါပြီ။
  - အရေးကြီးတာက `"development"` လို့ အတိအကျ သိခဲ့တဲ့ အသေးစိတ် အချက်အလက်လေး ပျောက်သွားတာပါ။ TypeScript က "ဒီ `env` က သုံးခုထဲက တစ်ခုခုပဲ ဖြစ်နိုင်တယ်။ ဘယ်ဟာလဲဆိုတာ အတိအကျတော့ မသိတော့ဘူး" လို့ ဖြစ်သွားပါတယ်။
- `annotatedConfig.featureFlags`: သူ့ရဲ့ Type က `Record<string, boolean>` ဖြစ်သွားပါတယ်။
  - ဒီမှာလည်း ပြဿနာက `darkMode` နဲ့ `newOnboarding` ဆိုတဲ့ Keys တွေ ပါတယ်ဆိုတာ TypeScript က မသိတော့ပါဘူး။ သူက "Key က စာသား ဖြစ်ရမယ်၊ Value က true/false ဖြစ်ရမယ်" လို့ပဲ ယေဘုယျ သိသွားပါပြီ။

### အဓိက ပြဿနာ (The Crux)

အကယ်၍ ကျွန်တော်တို့က `annotatedConfig.featureFlags.darkMode` လိုမျိုး အတိအကျ Access လုပ်ဖို့ ကြိုးစားရင် -

<Code lang="typescript" code={annotationProblem} title="config.ts" />

<Aside title="ရှင်းပြချက်" >
`AppConfig` ဆိုတဲ့ Type ကို `:` လေးနဲ့ သတ်မှတ်လိုက်တဲ့အတွက် ကျွန်တော်တို့က "ဒီ Object ဟာ `AppConfig` ရဲ့ စည်းမျဉ်းအတိုင်းပဲ" လို့ TypeScript ကို ပြောလိုက်တာပါပဲ။ TypeScript ကလည်း စည်းမျဉ်းအတိုင်း ဟုတ်မဟုတ်တော့ စစ်ပေးပါတယ်။ (ဥပမာ - `port` ကို စာလုံးပေါင်းမှားရင် ဒါမှမဟုတ် စာသား ထည့်မိရင် Error ပြပါလိမ့်မယ်)။ ဒါပေမဲ့ ဒီလို စစ်ပေးလိုက်တော့ `env` မှာ အတိအကျ `"development"` ဆိုတဲ့ အသေးစိတ် အချက်အလက်၊ ဒါမှမဟုတ် `featureFlags` ထဲမှာ `darkMode` ဆိုတဲ့ Key ပါတယ်ဆိုတဲ့ အသေးစိတ် အချက်အလက်တွေကို သူက `Record<string, boolean>` လိုမျိုး ယေဘုယျ Type အဖြစ် ပြောင်းလိုက်လို့ မသိတော့ပါဘူး။ ဒါကြောင့် အဲဒီ အသေးစိတ်တွေကို ခေါ်သုံးဖို့ ခက်သွားတာပေါ့ ခင်ဗျာ။
</Aside>

## တစ်စိတ်တစ်ပိုင်း ဖြေရှင်းနည်း: `as const`

ဒီပြဿနာကို ခဏ ဖြေရှင်းဖို့ `as const` ဆိုတာ သုံးလို့ ရပါတယ်။

<Code lang="typescript" code={asKeyword} title="config.ts" />

<Aside title="ရှင်းပြချက်">
  `as const` က "ဒီ Object က အတိအကျ ဒီအတိုင်းပဲ ဖြစ်နေမယ်။ ဘာမှ မပြောင်းလဲဘူး"
  လို့ ပြောလိုက်တာပါ။ ဒါကြောင့် `as const` လုပ်လိုက်ရင် TypeScript က အတိအကျ
  သိသွားပါတယ်။ ပြီးတော့ ဘယ်တော့မှ ပြင်လို့မရအောင် (Readonly) လည်း လုပ်ပေးပါတယ်။
  ဒါပေမဲ့ သူက `constConfig` ဟာ `AppConfig` ဆိုတဲ့ စည်းမျဉ်းနဲ့ ကိုက်ညီမှု
  ရှိမရှိကိုတော့ မစစ်ပေးပါဘူး။ ဥပမာ - `env` ကို တခြားစာလုံး `environment` လို့
  ရေးမိရင်တောင် `as const` လို့ သုံးထားရင် TypeScript က ဘာ Error မှ ပြမှာ
  မဟုတ်ပါဘူး ခင်ဗျာ။
</Aside>
