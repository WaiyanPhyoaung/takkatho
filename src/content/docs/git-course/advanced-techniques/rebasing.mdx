  `git rebase` ရဲ့ အခြေခံ သဘောတရားကို နားလည်ပြီး၊ Local History ကို ရှင်းလင်းတာနဲ့ Remote Branch က အပြောင်းအလဲတွေကို ထည့်သွင်းတာတွေမှာ ဘယ်လို အသုံးပြုရမလဲ ဆိုတာ လေ့လာနိုင်ဖို့ ဖြစ်ပါတယ်။ `git merge` နဲ့ ဘယ်လို ကွာခြားသလဲ ဆိုတာနဲ့ "Rebase ရဲ့ Golden Rule" ကိုပါ သိရှိနိုင်မှာပါ ခင်ဗျာ။
</Aside>

## `git rebase` ဆိုတာ ဘာလဲ? (ကိုယ့် Story ကို ပြန်ရေးခြင်း)

**git rebase ဆိုတာ -**
- Branch တစ်ခုက Commits တွေကို Commit အသစ်တစ်ခုအဖြစ် ပြန်လည် တည်ဆောက်တဲ့ နည်းလမ်းတစ်ခုပါ။
- ကိုယ့် Branch ရဲ့ စတင်တဲ့နေရာ (Base) ကို ပြောင်းလဲပစ်လိုက်တာ ဖြစ်ပါတယ်။
- ကိုယ်က အခုမှ တခြားနေရာကနေ အလုပ်စခဲ့သလိုမျိုး ဖြစ်သွားစေပါတယ်။
- အလွယ်ပြောရရင် Commit History ကို ပြန်လည် ရေးသားလိုက်တာပါ။

### အသုံးပြုပုံများ (စတင်လေ့လာသူများအတွက်)

1. **Local Feature Branch History ကို ရှင်းလင်းခြင်း**: Merge မလုပ်ခင်မှာ မှတ်တမ်းများနေတဲ့ Commits တွေကို ပိုအဓိပ္ပာယ်ရှိတဲ့ Commits နည်းနည်းအဖြစ် ပေါင်းစပ်နိုင်ပါတယ်။ (Interactive Rebase သုံးပြီး လုပ်ပါတယ်)။
2. **Parent Branch က Changes တွေကို ကိုယ့် Branch ထဲ ထည့်သွင်းခြင်း**: ကိုယ့် Feature Branch ကို အမြဲ Up-to-date ဖြစ်နေအောင် ကူညီပေးပြီး၊ နောက်ဆုံး Merge လုပ်တဲ့အခါ ပိုပြီး သပ်သပ်ရပ်ရပ် ဖြစ်စေပါတယ်။

## Rebase (Linear Story) vs. Merge (Preserves Divergence)

### Scenario 1: Feature Branch ကို `main` က Changes တွေနဲ့ Update လုပ်ခြင်း

**ပြဿနာ**: ကိုယ်က `main` ကနေ `my-feature` ဆိုပြီး Branch သစ်ခွဲခဲ့တယ်။ ကိုယ် အလုပ်လုပ်နေတုန်းမှာ `main` မှာ အရေးကြီးတဲ့ Updates တွေ ဝင်လာတယ်။ ကိုယ်က အဲ့ဒီ Updates တွေကို အခု လက်ရှိ လုပ်နေတဲ့ `my-feature` ထဲကို သပ်သပ်ရပ်ရပ် ထည့်သွင်းချင်တယ်။

**ဘယ်လို လုပ်မလဲ?**

**2. `my-feature` Branch ကို ပြန်ပြောင်းပါ**
<Code code={switchToFeature} title='Bash' />

**3. `main` ပေါ်မှာ Rebase လုပ်ပါ**
<Code code={gitRebaseMain} title='Bash' />

**ဘာဖြစ်သွားသလဲ?**

Git က `my-feature` နဲ့ သက်ဆိုင်တဲ့ Commits တွေကို ခဏ ဘေးဖယ်ထားလိုက်ပါတယ်။ အဲ့ဒီနောက် `main` က Update Commits တွေကို အရင်ယူပြီး၊ စောစောက ဖယ်ထားတဲ့ `my-feature` ရဲ့ Commits တွေကို `main` ရဲ့ အပေါ်မှာ တစ်ခုချင်းစီ ပြန်တင်ပေးလိုက်ပါတယ်။

<Aside type='caution' title='Conflicts'>
  Rebase လုပ်နေစဉ်မှာ Merge လုပ်သလိုမျိုးပဲ Conflicts တွေ ကြုံရနိုင်ပါတယ်။ ပြန်တင်နေတဲ့ Commit တစ်ခုချင်းစီအတွက် ဖြေရှင်းရပါမယ်။ ဖြေရှင်းပြီးရင် `git add <resolved file name>` လုပ်ပါ။ ပြီးရင် `git rebase --continue` လို့ ရိုက်ရပါမယ်။ (မလုပ်တော့ဘူးဆိုရင် `git rebase --abort` နဲ့ ရပ်တန့်နိုင်ပါတယ်)။
</Aside>

<Aside type='tip' title='အကျိုးကျေးဇူး'>
  အခုဆိုရင် `my-feature` Branch မှာ `main` ရဲ့ Updates တွေ အားလုံး ပါဝင်သွားပါပြီ။ ကိုယ်က `main` Updates တွေ ရပြီးမှ Feature ကို စခဲ့သလို ဖြစ်သွားလို့၊ နောက်ဆုံး `main` ထဲ Merge လုပ်တဲ့အခါ သပ်ရပ်တဲ့ **Fast-forward Merging** မျိုး ဖြစ်သွားမှာပါ ခင်ဗျာ။
</Aside>

### Scenario 2: Interactive Rebase (`git rebase -i`) ဖြင့် Local Commits များကို ရှင်းလင်းခြင်း

**ပြဿနာ**: ကိုယ့် Local `my-feature` Branch ပေါ်မှာ (Push မလုပ်ခင်) `"WIP"`, `"fix typo"`, `"another small fix"` လိုမျိုး အစီအစဉ်မကျတဲ့ Commits လေးတွေ အများကြီး ရှိနေတယ်။ Pull Request မတင်ခင်မှာ ဒါတွေကို သင့်တော်တဲ့ Commits ၁ ခု၊ ၂ ခုလောက်ပဲ ဖြစ်အောင် ရှင်းလင်းချင်တယ်။

**ဘယ်လို လုပ်မလဲ? (Interactive Rebase)**

```bash
git rebase -i HEAD~3
```
(ဒါက "လက်ရှိ နေရာကနေ နောက်ဆုံး Commits (၃) ခုကို Interactive Rebase နဲ့ ပြန်ပြင်မယ်" လို့ ဆိုလိုတာပါ)

**ဘာဖြစ်သွားသလဲ?**

Git က သတ်မှတ်ထားတဲ့ Text Editor ကို Commits စာရင်းနဲ့ ဖွင့်ပေးပါမယ်။ Commit တစ်ခုချင်းစီအတွက် အောက်ပါ လုပ်ဆောင်ချက်တွေကို ရွေးချယ်နိုင်ပါတယ် -

- **pick**: Commit ကို ရှိတဲ့အတိုင်း ဆက်သုံးမယ်။
- **reword (r)**: Commit ကို သုံးမယ်၊ ဒါပေမဲ့ Message ကို ပြင်ရေးမယ်။
- **squash (s)**: ဒီ Commit ရဲ့ Changes တွေကို အပေါ်က Commit ထဲ ပေါင်းထည့်မယ် (Message ကိုလည်း ပေါင်းစပ်မယ်)။
- **fixup (f)**: Squash နဲ့ ဆင်တူပေမဲ့၊ ဒီ Commit ရဲ့ Message ကိုတော့ ပယ်ဖျက်လိုက်မယ်။

<Aside type='tip' title='မှတ်ချက်'>
  အဓိကကတော့ `rebase -i` ကို သုံးပြီး Commits တွေကို ပြန်ပြင်ဆင်နိုင်တာပါ။ အသုံးများတာကတော့ Commits အသေး ၃ ခုလောက်ကို ၁ ခုတည်း ဖြစ်အောင် ပေါင်းလိုက်တာမျိုး (Squash/Fixup လုပ်တာမျိုး) ပါ။
</Aside>

### ဥပမာ

<ContentImage src={gitRebase} alt='rebasing'/>

```bash
pick 37709fc WIP
pick 3980a2f fix typo  # ဒါကို 's' သို့မဟုတ် 'squash' လို့ ပြောင်းပါ
pick f259ca8 actual feature work # 'pick' ကို 'r' သို့မဟုတ် 'reword' လို့ ပြောင်းပါ
```

<Aside type='tip' title='ရှင်းလင်းချက်'>
- **ပထမစာကြောင်း**: `WIP` ဆိုတဲ့ Commit ကို ဘာမှ မပြောင်းဘဲ ထားတာပါ။
- **ဒုတိယစာကြောင်း**: `fix typo` ဆိုတဲ့ Commit ကို မယူတော့ဘဲ၊ အပေါ်က (WIP) ထဲကို Changes တွေ ပေါင်းထည့်လိုက်တာပါ။
- **တတိယစာကြောင်း**: `actual feature work` ကိုတော့ ယူမယ်၊ ဒါပေမဲ့ Message ကို ပြန်ပြင် (reword) မယ်လို့ ပြောတာပါ။

Save လုပ်ပြီးတဲ့အခါ -
- Git က Commit အသစ်ရဲ့ Message ကို ပြန်ရေးဖို့ Editor တစ်ခု ထပ်ဖွင့်ပေးပါလိမ့်မယ်။ အဲ့ဒီမှာ WIP နဲ့ fix typo စာသားတွေ ပါနေတတ်ပြီး၊ နှစ်ခုပေါင်းပြီး ပိုရှင်းလင်းတဲ့ Message အသစ် ရေးပေးလို့ ရပါတယ်။
- ပြီးရင် reword အတွက်လည်း Editor ထပ်ပွင့်လာပြီး 'actual feature work' ဆိုတဲ့ Message ကို လိုသလို ပြန်ပြင်နိုင်ပါတယ် ခင်ဗျာ။
</Aside>

<Aside title='သတိပြုရန်' type='danger'>
  **Push ပြီးသား Shared History တွေကို လုံးဝ (လုံးဝ) Rebase မလုပ်ပါနဲ့။** 
  
  အဲ့ဒီ Shared Commits တွေအပေါ် မူတည်ပြီး အလုပ်လုပ်နေတဲ့ တခြားသူတွေ ရှိနေနိုင်လို့ပါ။ Rebase က History ကို ပြန်ပြင်ရေးတာ ဖြစ်လို့၊ ကိုယ့် Teammates တွေရဲ့ History နဲ့ လုံးဝ ကွဲလွဲပြီး ပြဿနာ အကြီးကြီး တက်နိုင်ပါတယ်။

  > **Push မလုပ်ရသေးတဲ့ ကိုယ်ပိုင် Local Branch မှာသာ Commits တွေ ရှုပ်နေရင် Rebase သုံးပါ ခင်ဗျာ။**
</Aside>
