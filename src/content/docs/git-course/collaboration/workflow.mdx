## အခြေခံ Workflow

လူအများကြီး ဝိုင်းပြီး Code တွေ ရေးကြတဲ့အခါ `main` Branch ထဲကို အကုန်လုံးက တိုက်ရိုက် `push` လုပ်ကြရင် အရမ်း ရှုပ်ထွေးသွားပြီး Code တွေ ပျက်စီးသွားတတ်ပါတယ်။ အဲဒီလို မဖြစ်ရအောင် အပြောင်းအလဲတွေကို အဖွဲ့လိုက် စနစ်တကျ တင်ပြဖို့နဲ့ ပေါင်းစပ်ဖို့ သေချာ စီမံထားတဲ့ နည်းလမ်းတစ်ခု လိုအပ်ပါတယ် ခင်ဗျာ။

### Feature Branch Workflow (ရိုးရှင်းသော "GitHub Flow" ပုံစံ)

ဒါက အဖွဲ့လိုက် အလုပ်လုပ်ဖို့အတွက် အရမ်း အသုံးများပြီး အကျိုးရှိတဲ့၊ နားလည်ရ လွယ်ကူတဲ့ နည်းလမ်းတစ်ခု ဖြစ်ပါတယ်။

**1. `main` ကနေ အမြဲ စတင်ပါ**
- ကိုယ့် Local `main` Branch က Remote `main` နဲ့ နောက်ဆုံး အခြေအနေ (Up-to-date) တူညီနေဖို့ အမြဲ သေချာပါစေ။
- `git switch main` သွားပြီး `git pull origin main` လို့ ရိုက်ပါ။

**2. Branch အသစ် တစ်ခု ဖန်တီးပါ**
- အလုပ်အသစ်တစ်ခု (Feature, Bug Fix, ဒါမှမဟုတ် စမ်းသပ်မှုတစ်ခု) လုပ်တော့မယ်ဆိုရင် `main` ကနေ Branch အသစ်တစ်ခု အမြဲ ခွဲထုတ်ပါ။
- `git switch -c <branch-name>`

**3. ကိုယ့်အလုပ်တွေကို အဲဒီ Branch ပေါ်မှာပဲ လုပ်ပါ**
- လိုအပ်တဲ့ ပြောင်းလဲမှုတွေ လုပ်ပြီး `commit` မှတ်ပါ။
- အသေးစား `commit` လေးတွေ များများ ခွဲပြီး မှတ်ဖို့ အကြံပြုပါတယ်။

**4. ကိုယ့် Branch ကို Push လုပ်ပါ (`main` ကို မဟုတ်ပါ!)**
- `git push -u origin <branch-name>`
- ( `-u` ကို ဒီ Branch အတွက် ပထမဆုံးအကြိမ် Remote ဆီ Push လုပ်တဲ့အခါ ထည့်သုံးဖို့ မမေ့ပါနဲ့)။

**5. `main` Branch က အမြဲတမ်း Clean ဖြစ်နေပါမယ်**
- ဒီ အဆင့်တွေ အားလုံးမှာ ကိုယ်က Remote ပေါ်က `main` Branch ကို တိုက်ရိုက် Push လုပ်လိုက်တာ လုံးဝ မရှိပါဘူး။ ကိုယ့် Branch နဲ့ ကိုယ် သီးသန့် အလုပ်လုပ်သွားတာ ဖြစ်ပါတယ်။ ပြီးမှ PR ကနေတဆင့် တရားဝင် တင်ပြရမှာပါ။

<Aside title="အနှစ်ချုပ်">
  ကိုယ့်အလုပ်တွေကို သီးခြား Branch (Separate Branches) တွေပေါ်မှာပဲ အမြဲ လုပ်ပါ။ အဲဒီ Branches တွေကိုပဲ Push လုပ်ပါ (`main` ကို တိုက်ရိုက် မ Push ရပါ)။ `main` ကို အမြဲတမ်း တည်ငြိမ် (Stable) အောင် ထားရှိပါ။ ဒါက ကိုယ့် အပြောင်းအလဲတွေကို အဖွဲ့ဆီ စနစ်တကျ တရားဝင် တင်ပြတဲ့ နည်းလမ်းကောင်း ဖြစ်ပါတယ် ခင်ဗျာ။
</Aside>
