ပြန်စစ်ခြင်း အတတ်ပညာ - ဆွေးနွေးခြင်း နှင့် ပိုကောင်းအောင် ပြုပြင်ခြင်း

(ခန့်မှန်းအချိန်: မိနစ် ၂၀)

ရည်ရွယ်ချက်: Pull Request ထဲမှာ code review (code ပြန်စစ်ခြင်း) လုပ်တဲ့ ရည်ရွယ်ချက်နဲ့ အခြေခံ အဆင့်ဆင့်ကို နားလည်ဖို့။

အကြောင်းအရာများ:

PR ဖွင့်ပြီးရင် ဘာဖြစ်လဲ? စကားဝိုင်း စပြီ!
ကိုယ့် Pull Request (PR) လေး ဖွင့်လိုက်ပြီဆိုတာနဲ့ Team members (အဖွဲ့ဝင်များ) ဒါမှမဟုတ် သတ်မှတ်ထားတဲ့ reviewers (ပြန်စစ်သူများ) ဆီကို အကြောင်းကြားစာ ရောက်သွားပါမယ်။ သူတို့က ကိုယ်လုပ်ထားတဲ့ changes (ပြောင်းလဲမှုများ) တွေကို review (ပြန်စစ်) ဖို့ စပါပြီ။ သူတို့က GitHub ပေါ်က PR ထဲမှာ ရှိတဲ့ "Files changed" tab (ပြောင်းထားတဲ့ ဖိုင်များ ကဏ္ဍ) ထဲကို ကြည့်ပြီး ကိုယ့် code တွေကို စစ်ဆေးကြပါလိမ့်မယ်။

ခေါင်းစဉ်: "တင်ပြထားသော Code ကို စစ်ဆေးခြင်း"

ပြန်စစ်ခြင်းရဲ့ အဓိက အချက်: Feedback (အကြံဉာဏ်) ပေးခြင်းနှင့် ဆွေးနွေးခြင်း။
ပြန်စစ်သူတွေက ဘာတွေ ကြည့်ကြလဲ (ရိုးရိုးလေး ပြောရရင်):

ကိုယ်ရေးထားတဲ့ code က ရည်ရွယ်ထားတဲ့အတိုင်း အလုပ်လုပ်ရဲ့လား?
သိသာထင်ရှားတဲ့ bugs (အမှားများ) တစ်ခုခု ပါနေလား?
code က ရှင်းရှင်းလင်းလင်း ရှိရဲ့လား၊ နားလည်ဖို့ လွယ်ရဲ့လား?
အဖွဲ့ရဲ့ team conventions (အဖွဲ့လိုက် လိုက်နာရမယ့် စည်းကမ်းများ) ဒါမှမဟုတ် best practices (အကောင်းဆုံး လုပ်နည်းများ) နဲ့ ကိုက်ညီရဲ့လား?
အခု လုပ်ထားတဲ့ နည်းထက် ပိုကောင်းတဲ့ တခြား လုပ်နည်းတွေ ရှိသေးလား?
Feedback (အကြံဉာဏ်) ပေးခြင်း (GitHub ပေါ်မှာ):

reviewers တွေက code ရဲ့ specific lines (သတ်မှတ်ထားတဲ့ စာကြောင်းများ) ပေါ်မှာ comments (မှတ်ချက်များ) ပေးထားလို့ ရပါတယ်။ သူတို့က မေးခွန်း မေးတာ ဒါမှမဟုတ် changes တွေ ဘယ်လို လုပ်သင့်တယ်ဆိုတာ အကြံပြုတာမျိုး လုပ်နိုင်တယ်။ reviewers တွေက PR ကို approve (အတည်ပြု) လုပ်လို့ ရသလို changes တွေ ထပ်လုပ်ဖို့ request (တောင်းဆို) တာမျိုးလည်း လုပ်နိုင်တယ်။

PR Author (PR ဖွင့်သူ) အနေနဲ့ ကိုယ့်ရဲ့ တာဝန်:

reviewers တွေ ပေးတဲ့ feedback ကို ဂရုတစိုက် ဖတ်ပါ။ comments တွေကို ပြန်ဖြေပါ၊ မေးခွန်းတွေကို ဖြေပေးပါ။ အကယ်၍ changes (ပြင်ဆင်မှုများ) လုပ်ဖို့ လိုအပ်တယ်ဆိုရင် ကိုယ့်ရဲ့ local feature branch ပေါ်မှာ ပြင်ဆင်ပါ၊ commit လုပ်ပါ၊ ပြီးတော့ အဲဒီ branch ကိုပဲ ပြန်ပြီး git push လုပ်လိုက်ပါ။ GitHub ပေါ်က ကိုယ့် PR က အလိုအလျောက် update (အသစ်ပြောင်း) သွားပါလိမ့်မယ်။ ဒါက စာတည်းက ကိုယ့် စာအုပ်အခန်းကို feedback ပြန်ပေးတာနဲ့ တူပါတယ်။ ကိုယ်က အဲဒီ feedback အတိုင်း ကိုယ့်အခန်းကို ပြန်ပြင်ပြီး updated version (အသစ် ပြင်ထားတဲ့ Version) ကို ပြန်တင်ပြလိုက်တာပေါ့။

ခေါင်းစဉ်: "Collaboration (အတူတူ အလုပ်လုပ်ခြင်း) လက်တွေ့: PR Feedback Loop"

သင်ခန်းစာ ၄.၃ အနှစ်ချုပ်: PR review (PR ပြန်စစ်ခြင်း) ဆိုတာ discussion (ဆွေးနွေးခြင်း) နဲ့ constructive feedback (အပြုသဘောဆောင်တဲ့ အကြံဉာဏ်များ) ကနေတဆင့် code quality (code အရည်အသွေး) ပိုကောင်းလာအောင် အတူတူ ဝိုင်းဝန်း လုပ်ဆောင်တာပဲ။ comments တွေအတွက် changes လုပ်ပြီးရင် ကိုယ့် branch ကို push updates လုပ်ဖို့ မမေ့ပါနဲ့။
