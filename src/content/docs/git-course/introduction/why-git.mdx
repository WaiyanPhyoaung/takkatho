ကျွန်တော်တို့ Code တွေ ရေးတဲ့အခါ၊ Project တွေ လုပ်တဲ့အခါမှာ စစချင်းတော့ အားလုံး အဆင်ပြေနေတတ်ပါတယ် ခင်ဗျာ။ ဒါပေမဲ့ Project တွေ တဖြည်းဖြည်း ကြီးလာတာနဲ့အမျှ (ဒါမှမဟုတ်) တခြားသူတွေနဲ့ အတူတူ လုပ်ရတဲ့အခါကျရင်တော့ ပြဿနာတွေ စပေါ်လာတတ်ပါတယ်။

### Git မသုံးခင် ကြုံရတာတွေ

- ကိုယ်ရေးနေတဲ့ Code ဖိုင်တွေ၊ Document ဖိုင်တွေကို `report_draft1.docx`, `report_final_v3.docx` စသဖြင့် နာမည်မျိုးစုံနဲ့ အကြိမ်ကြိမ် Save လုပ်ဖူးကြမှာပါ။ Code တွေနဲ့ဆို ပိုဆိုးပါတယ်။ တစ်ခုခု ပြောင်းလိုက်တိုင်း Version အသစ်ဆိုပြီး ထပ် Save ရတာမို့ မကြာခင်မှာပဲ Project Folder ကြီးက ရှုပ်ယှက်ခတ်သွားတတ်ပါတယ်။ ဘယ်ဟာက နောက်ဆုံး Version လဲ၊ ဘယ်ဟာက အဆင်ပြေနေတဲ့ Code လဲဆိုတာကို ခွဲခြားရ ခက်ခဲလာပါပြီ။

<ContentImage src={confusing} alt="confusing with file names" />

- ဒါ့အပြင် အရင်ရက်က ကောင်းကောင်း အလုပ်လုပ်နေတဲ့ Code က အခု တခြား အပြောင်းအလဲတွေ လုပ်လိုက်မှ ပျက်သွားတာမျိုးလည်း ကြုံဖူးကြမှာပါ။ အရင် Version တွေကို ပြန်ရှာရ၊ Copy-Paste လုပ်ရနဲ့ အချိန်တွေ အများကြီး ကုန်ရပါတယ်။

- Project အဆင်ပြေနေချိန်မှာ Feature အသစ်တစ်ခု ထပ်ထည့်ရမယ် (သို့) Code တွေကို အကြီးအကျယ် ပြန်ပြင်ရမယ်ဆိုရင် 'ရှိပြီးသားတွေပါ ပျက်သွားမလား' ဆိုပြီး စိုးရိမ်ရပါတယ်။ ဒီစိတ်က အလုပ်တွေကို နှေးကွေးစေပါတယ်။

### အဖွဲ့လိုက် အလုပ်လုပ်တဲ့အခါ ဖြစ်တတ်တဲ့ ပြဿနာတွေ

အဖွဲ့လိုက် အလုပ်လုပ်တဲ့အခါဆို ပိုဆိုးပါတယ်။ လူတစ်ယောက် ပြင်ထားတာကို နောက်တစ်ယောက်က ပြန်ထပ်ရေးမိလို့ ပျောက်သွားတာမျိုးတွေ ရှိတတ်ပါတယ်။ ဘယ်သူက၊ ဘယ်အချိန်မှာ၊ ဘာလို့ ပြောင်းလဲလိုက်တယ်ဆိုတဲ့ မှတ်တမ်း မရှိတဲ့အတွက် အဖွဲ့ဝင်တွေကြား နားလည်မှု လွဲတာတွေ ဖြစ်တတ်ပါတယ် ခင်ဗျာ။

### ကယ်တင်ရှင် ရောက်လာပြီ: Version Control စနစ်များ!

ဒီပြဿနာတွေ အားလုံးကို ဖြေရှင်းဖို့အတွက် Version Control System (VCS) ဆိုတဲ့ စနစ်တစ်ခု လိုအပ်လာတာပါ။

<Aside title="မှတ်ချက်">
  VCS ဆိုတာ ကိုယ့် Project ထဲက ဖိုင်တွေရဲ့ ပြောင်းလဲမှု မှတ်တမ်းတွေကို စနစ်တကျ သိမ်းဆည်းပေးတဲ့ Software ပါ။ ဘယ်သူက ဘာကို၊ ဘယ်အချိန်မှာ ပြောင်းလဲခဲ့တယ်ဆိုတာ အတိအကျ မှတ်ထားပေးပါတယ်။ ဒါကြောင့် ဘယ်အချိန်က Version ကိုမဆို အလွယ်တကူ ပြန်ကြည့်လို့ရသလို၊ အဖွဲ့လိုက် အလုပ်လုပ်တဲ့အခါလည်း ပြဿနာ မဖြစ်အောင် ကူညီပေးပါတယ်။
</Aside>

VCS ကို ဂိမ်းတွေဆော့တဲ့အခါ သုံးတဲ့ "Save Game" စနစ်နဲ့ တူတယ်လို့ စဥ်းစားကြည့်ပါ။ Project မှာ အရေးကြီးတဲ့ အဆင့်ရောက်တိုင်း Save (Commit) လုပ်ထားလို့ရပါတယ်။ မှားသွားရင်လည်း အရင် Save ထားတဲ့ အချိန်ကို အလွယ်တကူ ပြန်သွားလို့ရပါတယ်။ Code တွေကိုလည်း လမ်းကြောင်းခွဲပြီး (Branching) စမ်းသပ်လို့ ရပါသေးတယ်။

**"ဒါကြောင့် VCS ဆိုတာ ဖိုင်တွေကို Backup ထားတာတင် မဟုတ်ဘဲ၊ ကိုယ့် Project ရဲ့ ပြောင်းလဲတိုးတက်လာပုံ သမိုင်းကြောင်း တစ်ခုလုံးကို မှတ်တမ်းတင်ထားတာ ဖြစ်ပါတယ် ခင်ဗျာ။"**

### ဘာလို့ Git ကိုပဲ ရွေးချယ်သင့်တာလဲ?

VCS တွေ အများကြီး ရှိတဲ့ထဲမှာ Git ကတော့ အခုခေတ်မှာ လူတိုင်းနီးပါး သုံးနေတဲ့ စံသတ်မှတ်ချက် တစ်ခု ဖြစ်လာပါပြီ။ Git ဟာ Distributed ဖြစ်တဲ့အတွက် Developer တိုင်းရဲ့ ကွန်ပျူတာမှာ Project ရဲ့ သမိုင်းကြောင်း အပြည့်အစုံ ရှိနေပါတယ်။ ဒါကြောင့် Offline လည်း အလုပ်လုပ်လို့ရသလို၊ လုံခြုံမှုလည်း ပိုရှိပါတယ်။

<ContentImage src={gitConnect} alt="Git connect" />

Git ဟာ တော်တော်လေး မြန်ဆန်ပြီး၊ Project အသေးလေးတွေကနေ အကြီးကြီးတွေအထိ အဆင်ပြေပြေ သုံးနိုင်ပါတယ်။ Git ကို ကျွမ်းကျင်စွာ အသုံးပြုနိုင်ခြင်းဟာ ယနေ့ခေတ် Developer တစ်ယောက်အတွက် မရှိမဖြစ် လိုအပ်တဲ့ Skill တစ်ခု ဖြစ်ပါတယ် ခင်ဗျာ။

### Git နှင့် Online Services များ ကွာခြားပုံ

စတင် လေ့လာသူတွေအတွက် Git နဲ့ GitHub, GitLab, Bitbucket လို Service တွေ ကွာခြားပုံကို နားလည်ဖို့ လိုပါတယ်။ ရိုးရိုးလေး ပြောရရင် Git ဆိုတာ ကိုယ့် ကွန်ပျူတာထဲမှာ Code တွေရဲ့ ပြောင်းလဲမှုတွေကို ထိန်းချုပ်ပေးတဲ့ "Tool (ကိရိယာ)" သက်သက်သာ ဖြစ်ပါတယ်။

<ContentImage src={gitAndOthers} alt="Git & GitHub" />

<Aside title="မှတ်ချက်">
  GitHub, GitLab, Bitbucket တွေကတော့ အဲဒီ Git Project တွေကို Online ပေါ်မှာ သိမ်းထားပေးတဲ့ နေရာ (Platform) တွေပါ။ အင်တာနက်ပေါ်မှာ Project ကို Backup ထားဖို့နဲ့ တခြားသူတွေနဲ့ ဝိုင်းပြီး အလုပ်လုပ်ဖို့အတွက် ဒီ Service တွေကို သုံးတာ ဖြစ်ပါတယ်။
</Aside>

အနှစ်ချုပ်ရရင် ကိုယ့် ကွန်ပျူတာထဲမှာ Git ကို သုံးပြီး အလုပ်လုပ်တယ်။ ပြီးတော့ အဲဒီ အလုပ်တွေကို Online ပေါ်က GitHub လို နေရာမျိုးမှာ လှမ်းပြီး သိမ်းဆည်းတာ ဖြစ်ပါတယ် ခင်ဗျာ。
