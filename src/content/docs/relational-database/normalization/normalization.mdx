## Normalization ဆိုတာဘာလဲ။

Normalization ဆိုတာ ရှုပ်ထွေးနေတဲ့ data တွေကို စနစ်တကျဖြစ်အောင် ရှင်းလင်းစီစဉ်တဲ့ လုပ်ငန်းစဉ်တစ်ခုဖြစ်ပါတယ်။
သူ့ရဲ့ အဓိကရည်ရွယ်ချက်နှစ်ခုကတော့ Data ထပ်နေတာတွေ (Redundancy) ကို လျှော့ချဖို့နဲ့ Data မှန်ကန်တိကျမှု (Data Integrity) ကို မြှင့်တင်ဖို့ဖြစ်ပါတယ်။
ဒီလုပ်ငန်းစဉ်မှာ အဆင့်ဆင့်လိုက်နာရမယ့် စည်းမျဉ်းတွေ ပါဝင်ပြီး
အဲဒီစည်းမျဉ်းတွေအတိုင်း table တွေကို စနစ်တကျ ဖွဲ့စည်းတည်ဆောက်ရပါတယ်။

## Normalization ဘာကြောင့် လိုအပ်တာလဲ။
လုပ်ငန်းတစ်ခုရဲ့ သတင်းအချက်အလက်အားလုံးကို table ကြီးတစ်ခုတည်းမှာ အားလုံးရောထည့်သိမ်းထားတယ်ဆိုပါစို့။
customer အချက်အလက်၊ သူတို့ရဲ့ order တွေ၊ ကုန်ပစ္စည်းအချက်အလက်တွေ အကုန်လုံးကို တစ်နေရာတည်းမှာ စုစည်းထားတာမျိုးပေါ့။ 
ဒါက အစပိုင်းမှာတော့ လွယ်ကူပုံရပေမယ့် လုပ်ငန်းကြီးလာတာနဲ့အမျှ data တွေကို စီမံခန့်ခွဲရတာ အရမ်းခက်ခဲပြီး ဆိုးဝါးတဲ့အခြေအနေကို ရောက်သွားနိုင်ပါတယ်။

ဒီလိုအခြေအနေမျိုးမှာ **Normalization** က အောက်ပါပြဿနာတွေကို ဖြေရှင်းပေးပါတယ်။

### 1. Data Redundancy (မလိုအပ်ဘဲ ထပ်နေတဲ့ Data တွေ) ကို ဖယ်ရှားခြင်း။

Data Redundancy ဆိုတာ တူညီတဲ့အချက်အလက်တစ်ခုကို နေရာများစွာမှာ ထပ်ခါထပ်ခါ သိမ်းဆည်းထားတာကို ဆိုလိုပါတယ်။ 
ဥပမာ- customer တစ်ယောက်က order ၁၀ ခုမှာထားရင်၊ သူ့ရဲ့လိပ်စာကို order ၁၀ ခုလုံးအတွက် ၁၀ ခါ ထပ်ပြီး သိမ်းထားတာမျိုး။

**ဘာပြဿနာရှိလဲ။**
- **နေရာကုန်တယ်:** မလိုအပ်ဘဲ storage space တွေ အများကြီးကုန်ကျစေပါတယ်။
- **ပြင်ဆင်ရခက်တယ်:** အဲဒီ customer က လိပ်စာပြောင်းသွားရင် နေရာ ၁၀ ခုလုံးမှာ လိုက်ပြင်ရပါမယ်။ တစ်နေရာပဲပြင်ပြီး ကျန်ခဲ့ရင် data တွေလွဲကုန်ပြီး ဘယ်ဟာအမှန်လဲဆိုတာ မသိနိုင်တော့ပါဘူး။

### 2. Data Anomalies (Data ပြဿနာများ) ကို တားဆီးခြင်း။
Data တွေကို စနစ်တကျမဖွဲ့စည်းထားတဲ့အတွက် အချက်အလက်တွေ ထည့်သွင်းတဲ့အခါ၊ ပြင်ဆင်တဲ့အခါ ဒါမှမဟုတ် ဖျက်ပစ်တဲ့အခါ ပြဿနာတွေ ပေါ်ပေါက်လာတတ်ပါတယ်။ အဓိကအားဖြင့် ပြဿနာသုံးမျိုးရှိပါတယ်။

- **Insertion Anomaly -**  အချက်အလက်တစ်ခုကို ထည့်ချင်ပေမယ့် ဆက်စပ်နေတဲ့ တခြားအချက်အလက်မရှိသေးလို့ ထည့်လို့မရတဲ့ အခြေအနေပါ။ <br />
ဥပမာ - order မမှာသေးတဲ့ customer အသစ်တစ်ယောက်ရဲ့ အချက်အလက်ကို Database ထဲမှာ မှတ်တမ်းတင်ချင်ပေမယ့် Table ကြီးတစ်ခုတည်းမှာ သိမ်းထားတဲ့အတွက် "order အချက်အလက်" ဖြည့်စရာမရှိလို့ 
customer အသစ်ကို ထည့်သွင်းလို့မရတာမျိုး ဖြစ်တတ်ပါတယ်။

- **Update Anomaly -** အပေါ်မှာပြောခဲ့တဲ့ Redundancy ကနေ တိုက်ရိုက်ဖြစ်ပေါ်လာတဲ့ ပြဿနာပါ။ <br />
ဥပမာ - customer တစ်ယောက်က လိပ်စာပြောင်းသွားလို့ သူ့ရဲ့ order မှတ်တမ်းတစ်ခုမှာ လိပ်စာအသစ်ကို ပြင်လိုက်ပေမယ့် ကျန်တဲ့ order အဟောင်းတွေမှာ လိပ်စာအဟောင်းအတိုင်း ကျန်နေခဲ့ရင်
အဲဒီ customer မှာ လိပ်စာအဟောင်းနဲ့ အသစ် နှစ်မျိုးလုံးရှိနေပြီး Data တွေ ကိုက်ညီမှုမရှိတော့ပါဘူး။

- **Deletion Anomaly -** အချက်အလက်တစ်ခုကို ဖျက်လိုက်တဲ့အခါ မရည်ရွယ်ဘဲ တခြားအရေးကြီးတဲ့ အချက်အလက်တွေပါ အတူတူပါသွားတာမျိုးပါ။ <br />
ဥပမာ - customer တစ်ယောက်မှာ order တစ်ခုပဲရှိပြီး အဲဒီ order ကို ဖျက်လိုက်တယ်ဆိုပါစို့။ Table ကြီးတစ်ခုတည်းမှာ သိမ်းထားတဲ့အတွက် order ကိုဖျက်လိုက်တာနဲ့ အဲဒီ Row တစ်ခုလုံးပျက်သွားပြီး 
customer ရဲ့ နာမည်၊ ဖုန်းနံပါတ်၊ လိပ်စာ စတဲ့ အချက်အလက်အားလုံးပါ Database ထဲကနေ လုံးဝပျောက်သွားနိုင်ပါတယ်။

Normalization လုပ်ခြင်းက database ကို ပိုမိုတိကျမြန်ဆန်စေပြီး ဒီလိုပြဿနာတွေမဖြစ်အောင် ကာကွယ်ပေးပါတယ်။
