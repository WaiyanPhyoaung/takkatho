## Candidate Key ဆိုတာ ဘာလဲ?

**Candidate Key** ဆိုတာ Table ထဲရှိ Record တစ်ခုချင်းစီကို တိတိကျကျ သီးသန့် ခွဲခြားပေးနိုင်တဲ့ Field (သို့မဟုတ် Field အစုအဝေး) ဖြစ်ပါတယ်။

"Candidate" (ကိုယ်စားလှယ်လောင်း) ဆိုတဲ့ နာမည်အတိုင်းပဲ၊ ဒီ Key တွေဟာ **Primary Key အဖြစ် ရွေးချယ်ခံရဖို့ အရည်အချင်း ပြည့်မီနေကြတဲ့ Key များ** ဖြစ်ပါတယ်။ Table တစ်ခုမှာ Candidate Key တစ်ခုထက်မက ရှိနိုင်ပါတယ်။

<Aside title="💡 လက်တွေ့ ဥပမာ">
| student_id | nrc_number | name | email |
|---|---|---|---|
| 1 | 12/LMN(N)123456 | Mg Mg | mgmg@gmail.com |
| 2 | 12/LMN(N)654321 | Aung Aung | aungaung@gmail.com |

ဒီ Table မှာဆိုရင်:
- `student_id` (ကျောင်းသားနံပါတ်) သည် လူတစ်ယောက်ချင်းစီအတွက် Unique ဖြစ်ပါတယ်။
- `nrc_number` (မှတ်ပုံတင်နံပါတ်) သည်လည်း တစ်ဦးချင်းစီအတွက် Unique ဖြစ်ပါတယ်။
- `email` (အီးမေးလ်) သည်လည်း Unique ဖြစ်ပါတယ်။

ဒါကြောင့် `student_id`, `nrc_number`, နဲ့ `email` သုံးခုစလုံးဟာ Primary Key အဖြစ် ရွေးချယ်ခံပိုင်ခွင့် ရှိကြတဲ့ **Candidate Keys** တွေ ဖြစ်ကြပါတယ်။ ဒီအထဲကမှ အသင့်တော်ဆုံး `student_id` ကို Primary Key အဖြစ် ရွေးချယ်လိုက်ပါမယ်။
</Aside>

## Candidate Key ရဲ့ မဖြစ်မနေ စည်းမျဉ်းများ

Field တစ်ခုဟာ Candidate Key ဖြစ်ဖို့ဆိုရင် အောက်ပါ စည်းမျဉ်းများနှင့် ကိုက်ညီရပါမယ်:

1. **Unique ဖြစ်ရမည်:** Candidate Key Value သည် တခြား Record များနှင့် ထပ်တူညီ၍ မရပါ။
2. **NOT NULL ဖြစ်ရမည်:** Value လွတ်နေခြင်း (`Null`) လုံးဝ ခွင့်မပြုပါ။
3. **Immutability (မကြာခဏ မပြောင်းလဲရပါ):** Candidate Key Value သည် အချိန်နဲ့အမျှ ခဏခဏ ပြောင်းလဲနေတဲ့ Data မျိုး မဖြစ်သင့်ပါ။ (ဥပမာ - ဖုန်းနံပါတ်သည် ပြောင်းလဲနိုင်သဖြင့် Candidate Key အဖြစ် မသင့်တော်ပါ။)

## Artificial Candidate Key (Surrogate Key)

တစ်ခါတလေမှာ Table ထဲက ဘယ် Natural Field မှ (Unique မဖြစ်လို့) Candidate Key အဖြစ် သုံးလို့ မရတာမျိုး ကြုံရတတ်ပါတယ်။ (ဥပမာ - `cars` Table မှာ ကားအမျိုးအစားနဲ့ အရောင်လောက်ပဲ ပါတဲ့အခါမျိုး)

ဒီလို အခြေအနေမျိုးမှာ System ကနေ အလိုအလျောက် တိုးပေးမယ့် Auto-increment `id` (ဥပမာ - 1, 2, 3, 4) ဆိုတဲ့ Field အသစ်တစ်ခု ဖန်တီးပြီး Candidate/Primary Key အဖြစ် သုံးကြပါတယ်။ ဒါကို **Surrogate Key (Artificial Candidate Key)** လို့ ခေါ်ဆိုပါတယ်။
