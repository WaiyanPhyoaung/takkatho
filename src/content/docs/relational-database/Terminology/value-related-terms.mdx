## Data vs Information
Database နဲ့ပတ်သက်တဲ့အခါ မကြာခဏ အချင်းချင်းဖလှယ်ပြီး သုံးတတ်ကြပေမယ့် တိကျတဲ့အဓိပ္ပာယ်တွေ ရှိကြတဲ့ အခြေခံသဘောတရား နှစ်ခုနဲ့စလိုက်ရအောင်။

### အချက်အလက် - Data
Data ဆိုသည်မှာ database တစ်ခုအတွင်း သိမ်းဆည်းထားသော မပြုပြင်ရသေးသည့်၊ လုပ်ဆောင်ခြင်းမပြုရသေးသည့် အချက်အလက်များဖြစ်ပါတယ်။ 
ဒါတွေက ကိန်းဂဏန်း၊ ရက်စွဲ၊ နာမည်၊ စာသား ဒါမှမဟုတ် တိုင်းတာနိုင်သည့် တန်ဖိုးတစ်ခုခု ဖြစ်နိုင်ပါတယ်။ 
တိတိကျကျ ပြင်ဆင်မွမ်းမံခြင်း မလုပ်မချင်း သိမ်းဆည်းထားတဲ့ အခြေအနေအတိုင်းပဲ ရှိနေပါတယ်။

ဥပမာ - 
အောက်ပါ row of data ကို စဉ်းစားကြည့်ပါ။

Alice&nbsp;&nbsp;&nbsp;&nbsp;2480284&nbsp;&nbsp;&nbsp;&nbsp;05/09/2005&nbsp;&nbsp;&nbsp;&nbsp;85.00

ဒီ row ကို ကြည့်လိုက်ရင် ကျွန်တော်တို့ ဘာမှသိပ်မသိနိုင်ပါဘူး။ 
- 2480284 ဆိုတာ ဘာကို ဆိုလိုတာလဲ။ customer ID လား၊ product code လား၊ ဒါမှမဟုတ် တခြားတစ်ခုခုလား။ Alice နဲ့ ဆက်စပ်မှု ရှိသလား။
- 85.00 ကရော ဘာကို ကိုယ်စားပြုတာလဲ။ ရမှတ် (score) လား၊ ငွေပေးချေမှု (payment) လား၊ ရာခိုင်နှုန်း (percentage) လား။
အကြောင်းအရာ (context) မရှိဘဲနဲ့ ဒါတွေဟာ အချက်အလက်(raw data) သက်သက်ပဲ ဖြစ်ပါတယ်။ 
အဲဒီအချက်အလက်တွေကို စနစ်တကျ စီမံခန့်ခွဲပြီး (process) မှသာ သူတို့ရဲ့ အဓိပ္ပာယ်ကို နားလည်နိုင်မှာ ဖြစ်ပါတယ်။

### သတင်းအချက်အလက် - Information (အဓိပ္ပာယ်ရှိသော ဒေတာ)

သတင်းအချက်အလက် (Information) ဆိုတာကတော့ သင်အသုံးပြုတဲ့အခါ ဒါမှမဟုတ် ကြည့်ရှုတဲ့အခါမှာ အဓိပ္ပာယ်ရှိပြီး အသုံးဝင်အောင် ပြုပြင်စီမံထားတဲ့ ဒေတာ (Data) တွေပဲဖြစ်ပါတယ်။
အဓိက မှတ်သားထားရမှာကတော့ ကျွန်တော်တို့ဟာ ဒေတာတွေကို အဓိပ္ပာယ်ရှိတဲ့ သတင်းအချက်အလက်အဖြစ် ပြောင်းလဲနိုင်ဖို့ တစ်နည်းနည်းနဲ့ စီမံဆောင်ရွက်ရမယ် ဆိုတာပါပဲ။
<br />
အဲဒီ "Alice&nbsp;&nbsp;&nbsp;&nbsp;2480284&nbsp;&nbsp;&nbsp;&nbsp;05/09/2005&nbsp;&nbsp;&nbsp;&nbsp;85.00" ဆိုတဲ့ ဒေတာကို "ကျောင်းသားစာရင်း" ဆိုတဲ့ table ထဲမှာ ထည့်လိုက်ပြီး၊
"ကျောင်းသားအမည် (Student Name)"၊ "ကျောင်းသား ID (Student ID)"၊ "မွေးနေ့ (Date of Birth)"၊ "ရမှတ် (Score)" စတဲ့ ခေါင်းစဉ်တွေနဲ့ စနစ်တကျ ထားလိုက်တယ်ဆိုပါစို့။
အဲဒီအခါမှာ "Alice" က ကျောင်းသားနာမည်၊ "2480284" က ကျောင်းသား ID၊ "05/09/2005" က မွေးနေ့၊ "85.00" က ရမှတ်ဆိုပြီး အဓိပ္ပာယ်ရှင်းသွားပါပြီ။
ဒီလို အဓိပ္ပာယ်ရှင်းသွားတဲ့အခါ "Alice ဆိုတဲ့ ကျောင်းသားဟာ ID နံပါတ် 2480284 ရှိပြီး 2005 ခုနှစ် မေလ ၉ ရက်နေ့မှာ မွေးတယ်။ စာမေးပွဲမှာ ၈၅ မှတ်ရခဲ့တယ်" ဆိုတဲ့ အသုံးဝင်တဲ့ သတင်းအချက်အလက်တစ်ခု ဖြစ်သွားပါပြီ။
<br />
အချက်အလက် (Data) နဲ့ သတင်းအချက်အလက် (Information) တို့ရဲ့ ကွာခြားချက်ကို နားလည်ထားဖို့ အရေးကြီးပါတယ်။ 
database တွေဆိုတာကလည်း raw data တွေကို သိမ်းဆည်းဖို့သက်သက် မဟုတ်ဘဲ၊ လုပ်ငန်း ဒါမှမဟုတ် အဖွဲ့အစည်းအတွက် အဓိပ္ပာယ်ရှိပြီး အသုံးဝင်တဲ့ သတင်းအချက်အလက်(information)တွေ ပြန်ထုတ်ပေးနိုင်ဖို့အတွက် ဒီဇိုင်းဆွဲထားရန် လိုအပ်ပါတယ်။

>အချက်အလက် (Data) ဆိုတာ ကျွန်တော်တို့သိမ်းဆည်းထားတဲ့ အရာဖြစ်ပါတယ်။
>
>သတင်းအချက်အလက် (Information) ဆိုတာ ကျွန်တော်တို့ပြန်လည်ထုတ်ယူရရှိတဲ့ အရာဖြစ်ပါတယ်။

## Null

Null ဆိုတာ data မရှိတဲ့၊ ပျောက်နေတဲ့ ဒါမှမဟုတ် မသိရသေးတဲ့ တန်ဖိုးတစ်ခုကို ကိုယ်စားပြုတာပါ။ Null ဟာ သုည (zero) လည်း မဟုတ်ပါဘူး၊ နေရာလွတ် (blank) လည်း မဟုတ်ပါဘူး၊ 
ဗလာ (empty) လည်း မဟုတ်ပါဘူး။ တကယ်ကို မရှိတာ (truly absent) ကို ဆိုလိုတာဖြစ်ပါတယ်။

### နားလည်မှုအလွဲများ

**0 (zero)**

Zero ဟာ အဓိပ္ပာယ်အမျိုးမျိုး ရှိနိုင်ပါတယ်။ ဥပမာ - ဘဏ်အကောင့်လက်ကျန် 0 (balance = 0) ဖြစ်တာ၊ ကုန်ပစ္စည်းစတော့ 0 (stock level = 0) ဖြစ်တာမျိုးကို ကိုယ်စားပြုနိုင်ပါတယ်။ ဆိုလိုတာက 
0 ဟာ "မရှိဘူး" လို့ ပြောတာမျိုး မဟုတ်ဘဲ "ပမာဏက 0" ဖြစ်တယ်လို့ တိတိကျကျ ဖော်ပြတာဖြစ်ပါတယ်။

**Blank Space**

နေရာလွတ် (space) တစ်ခု ဒါမှမဟုတ် တစ်ခုထက်ပိုတဲ့ space တွေဟာ ကျွန်တော်တို့အများစုအတွက် အဓိပ္ပာယ်မရှိဘူးလို့ ထင်ရပါတယ်။ ဒါပေမဲ့ SQL လိုမျိုး query language တွေအတွက်တော့ အဓိပ္ပာယ်ရှိပါတယ်။ 
SQL က blank space ဆိုတာကို မှန်ကန်တဲ့ character (စာလုံး) တစ်ခုအဖြစ် လက်ခံပါတယ်။ ဒါဟာ "တန်ဖိုးမရှိတာ" မဟုတ်ဘဲ "နေရာလွတ်" ဆိုတဲ့ တန်ဖိုးရှိနေတာဖြစ်ပါတယ်။

**Zero-length String**

စာလုံးလုံးဝမပါဝင်တဲ့ string (single quote နှစ်ခုကြားထဲမှာ space လုံးဝမပါတာ) ဟာလည်း SQL လို ဘာသာစကားတွေအတွက် လက်ခံနိုင်တဲ့ တန်ဖိုးတစ်ခုပါပဲ။ ဒါဟာ "စာလုံးမပါတဲ့ string" ဆိုတဲ့ တန်ဖိုးကို ပိုင်ဆိုင်ပါတယ်။ Null လို "မရှိ" တာမျိုး မဟုတ်ပါဘူး။


> **Null ကတော့ အချက်အလက် မရှိခြင်း/မသိရှိခြင်း ကိုကိုယ်စားပြုပါတယ်။** 

ဥပမာ - customer တစ်ယောက်ရဲ့ ဖုန်းနံပါတ်ကို မဖြည့်ထားဘူးဆိုရင် အဲဒီနေရာမှာ Null သုံးနိုင်ပါတယ်။ 
ဒါဟာ "ဖုန်းနံပါတ် သုည" လည်း မဟုတ်ပါဘူး၊ "ဖုန်းနံပါတ် နေရာလွတ်" လည်း မဟုတ်ပါဘူး၊ "ဖုန်းနံပါတ် မဖြည့်ထား" တာဖြစ်ပါတယ်။ <br />

> ဒါကြောင့် Null, 0, ' ' နဲ့ '' တို့ဟာ မတူညီတဲ့ အဓိပ္ပာယ်တွေ ရှိကြပါတယ်။ Null ဟာ အချက်အလက်မရှိတာ၊ မသိသေးတာကို ကိုယ်စားပြုပြီး ကျန်တာတွေကတော့ တန်ဖိုးတစ်ခုခု ရှိနေတာကို ကိုယ်စားပြုပါတယ်။


### Null အမျိုးအစားများ
တန်ဖိုးတစ်ခုက ဘာကြောင့် Null ဖြစ်နေရတာလဲဆိုတာကို နားလည်ထားတာက data အရည်အသွေးနဲ့ ခွဲခြမ်းစိတ်ဖြာမှုအတွက် အရေးကြီးပါတယ်။

#### ၁။ ပျောက်နေသော တန်ဖိုးများ (Missing Values)
အများအားဖြင့် မေ့လျော့တာ၊ ဒေတာထည့်သွင်းစဉ် လူရဲ့အမှားအယွင်းကြောင့် ဒါမှမဟုတ် ဒေတာစုဆောင်းတဲ့ လုပ်ငန်းစဉ် မပြည့်စုံတာတွေကြောင့် ဖြစ်တတ်ပါတယ်။ 

ဥပမာ - ဝန်ထမ်းတစ်ယောက်က Customer တစ်ယောက်ရဲ့ ကိုယ်ရေးအချက်အလက်ကို စာရင်းသွင်းတုန်းက Email Address ကို ထည့်ဖို့ မေ့သွားတာကြောင့် အဲဒီ Email field မှာ Null ဖြစ်နေတာမျိုး။

#### ၂။ မသိသေးသော တန်ဖိုးများ (Unknown Values)
တကယ်တမ်းမှာ တန်ဖိုးတစ်ခု ရှိပေမဲ့ data record(row) တစ်ခုကို ဖန်တီးချိန် ဒါမှမဟုတ် ပြင်ဆင်(update) တဲ့အချိန်မှာ မသိသေးတာ ဒါမှမဟုတ် မရရှိသေးတာကို ဆိုလိုပါတယ်။

ဥပမာ - Project အသစ်တစ်ခု စတင်လိုက်ပြီး Budget တစ်ခု သတ်မှတ်ပေးမှာ သေချာပေမယ့် ဘယ်လောက်ဆိုတာကတော့ တိတိကျကျ အတည်ပြုရသေးတဲ့အခါ Budget Field ကို လောလောဆယ် NULL လို့ ထားပြီး နောက်မှ ဖြည့်တာမျိုး။


### တွက်ချက်မှုများနှင့် query များတွင် Null ၏ အခန်းကဏ္ဍ
Database တွေက Null တွေကို ကိုင်တွယ်တဲ့အခါမှာ မတူညီတဲ့စည်းကမ်းတွေ ရှိပါတယ်။ အထူးသဖြင့် သင်္ချာဆိုင်ရာဂဏန်းတွက်ချက်မှုတွေ (mathematical operations) နဲ့ အချက်အလက်မေးမြန်းထုတ်ယူမှုတွေ (queries) မှာ ဖြစ်ပါတယ်။ 

**Null တွေပါဝင်နေတဲ့ တန်ဖိုးတွေကို သင်္ချာနည်းအရ တွက်ချက်ကြည့်မယ်ဆိုရင် (ဥပမာ- ပေါင်းတာ ဒါမှမဟုတ် ပျမ်းမျှရှာတာ) ရလဒ်က null ဖြစ်သွားနိုင်သလို၊ null တွေကို သေချာဂရုစိုက်ပြီး မကိုင်တွယ်ဘူးဆိုရင် မှားယွင်းတဲ့ ရလဒ်တွေ ထွက်လာနိုင်ပါတယ်။**

| Operation | Result |
| ---- | ---- |
| `NULL + 1` | `NULL` | 
| `5 > NULL` | `NULL` | 
| `NULL = NULL` | `NULL` (not true) |
| `NULL IS NULL` | `TRUE` |

`NULL = NULL → NULL` (မသိရသေးတဲ့ တန်ဖိုးနှစ်ခုဟာ တူညီလား၊ မတူညီဘူးလားဆိုတာ ကျွန်တော်တို့ မသိနိုင်လို့ပါ။) <br />
`NULL > 5 → NULL` (NULL ကဘာလဲဆိုတာ ကျွန်တော်တို့ အတိအကျမသိနိုင်တဲ့အတွက် 5 နဲ့ မနှိုင်းယှဥ်နိုင်လို့ပါ။) <br />

NULL ပါဝင်တဲ့ မည်သည့် သင်္ချာတွက်ချက်မှုမဆို NULL ကိုပဲ ပြန်ပေးပါတယ်။ 

`10 + NULL → NULL` <br />
`'Hello' || NULL → NULL` <br />
ဘာကြောင့်လဲဆိုတော့
ကျွန်တော်တို့ဟာ သိရှိပြီးသား တန်ဖိုးတစ်ခု (known value) နဲ့ မသိရှိရသေးတဲ့ တန်ဖိုးတစ်ခု (unknown value) ကို ပေါင်းစပ်လိုက်တဲ့အခါ၊ ရလဒ်ကလည်း မသိရှိနိုင်တဲ့ တန်ဖိုးအဖြစ်ပဲ ဆက်လက်တည်ရှိနေလို့ပဲ ဖြစ်ပါတယ်။

#### NULL in Aggregate Functions
| Function | Behavior with NULLs |
| ---- | ---- |
| SUM(column) | Ignores NULLs | 
| AVG(column) | Ignores NULLs | 
| COUNT(column) | Counts only non-null values |
| COUNT(*) | Counts all rows, including NULLs |

တချို့တန်ဖိုးတွေ မသိရှိရသေးရင်တောင်မှ၊ သိရှိပြီးသား တန်ဖိုးတွေကိုတော့ aggregate လုပ်ဆောင်နိုင်ပါတယ်။

#### ဥပမာအနေနဲ့ ကြည့်ရအောင်။
| Name | Salary |
| ---- | ---- |
| Alice | 1000 | 
| Bob | 1200 | 
| Charlie | null |
| Brown | 900 |

ဝန်ထမ်းတွေရဲ့ လစာတွေကို မှတ်တမ်းတင်ထားတဲ့ table တစ်ခုရှိတယ်ဆိုပါစို့။ ဝန်ထမ်းအသစ်တစ်ယောက်ကို ခန့်ထားပေမယ့် လစာကို လောလောဆယ် မသတ်မှတ်ရသေးလို့ သူ့လစာ field မှာ Null ဖြစ်နေနိုင်ပါတယ်။

ဒီ table ထဲက ဝန်ထမ်းအားလုံးရဲ့ ပျမ်းမျှလစာကို ရှာမယ် ဆိုပါစို့။ (SQL မှာ AVG() ဆိုတဲ့ function ကို သုံးပြီး ရှာလို့ရပါတယ်။ ဥပမာ- SELECT AVG(Salary) FROM Employees;)

ဒီလိုတွက်ချက်တဲ့အခါ Null ဖြစ်နေတဲ့ လစာကို ထည့်မတွက်ပါဘူး။ ဂဏန်းတန်ဖိုးရှိတဲ့ လစာတွေဖြစ်တဲ့ 1000, 1200, 900 တို့ကိုပဲ ပေါင်းပြီး ဝန်ထမ်း ၃ ယောက်အတွက်ပဲ ပျမ်းမျှရှာမှာ ဖြစ်ပါတယ်။

(1000 + 1200 + 900) / 3 = 1033.33

ဒါကြောင့် Null တွေကို ဘယ်လို ကိုင်တွယ်မလဲဆိုတာက အချက်အလက်တွေ မှန်ကန်တိကျဖို့နဲ့ ဆုံးဖြတ်ချက်တွေ ချမှတ်ရာမှာ အရေးပါပါတယ်။ 
သင်္ချာတွက်ချက်မှုတွေနဲ့ query တွေ မလုပ်ခင်မှာ ဘယ် field တွေက Null ဖြစ်နိုင်လဲ၊ Null ဖြစ်ရင် ဘယ်လို ကိုင်တွယ်မလဲဆိုတာကို သေချာ စဉ်းစားထားဖို့ လိုအပ်ပါတယ်။ (ဥပမာ- Null တွေကို 0 နဲ့ အစားထိုးမလား၊ Null တွေကိုထည့်မတွက်ပဲ လုံးဝချန်လှပ်မလား စသည်ဖြင့်)

### Null အတွက် ဒီဇိုင်းဆိုင်ရာ ထည့်သွင်းစဉ်းစားချက်များ

Database ကို ဒီဇိုင်းဆွဲရာတွင် filed တစ်ခုစီအတွက် Null တွေရဲ့ သက်ရောက်မှုတွေကို ထည့်သွင်းစဉ်းစားဖို့ မရှိမဖြစ်လိုအပ်ပါတယ်။

- ဒီ field က တကယ်ပဲ Null ဖြစ်နိုင်သလား။ (ဥပမာ- customer တစ်ယောက်မှာ ဖုန်းနံပါတ်မရှိတာမျိုး ဖြစ်နိုင်သလား။)
- ဒီ field ကို Null တန်ဖိုးလက်ခံဖို့ ခွင့်ပြုသင့်သလား။ (ဥပမာ- ကုန်ပစ္စည်းကုတ်နံပါတ် (product ID) က Null ဖြစ်လို့ ရသလား။)
- Null တွေ ပါဝင်နေတာက သင့်ရဲ့ query တွေ၊ တွက်ချက်မှုတွေ ဒါမှမဟုတ် report တွေအပေါ် ဘယ်လို သက်ရောက်မှု ရှိနိုင်လဲ။ (ဥပမာ- AVG() function ကို သုံးရင် တချို့တန်ဖိုးတွေ Null ဖြစ်နေရင် မျှော်လင့်ထားတဲ့ ရလဒ် ထွက်လာမှာလား။)

ပျောက်နေတဲ့/မရှိတဲ့ တန်ဖိုးတွေ (Missing Values)၊ မသိသေးတဲ့ တန်ဖိုးတွေ (Unknown Values) နဲ့ သင်္ချာတွက်ချက်မှုတွေ ဒါမှမဟုတ် Aggregate Function တွေလုပ်ဆောင်ရာတွင် null သက်ရောက်မှုတွေကို သေချာ ထည့်သွင်းစဉ်းစားတာက Database Design ရဲ့ အရေးကြီးတဲ့ အစိတ်အပိုင်းတစ်ခု ဖြစ်ပါတယ်။


<Aside title="အနှစ်ချုပ်">
- NULL ဆိုတာ "မသိရှိရသေးတဲ့" သဘောကို ဆောင်ပါတယ်။ ဒါဟာ တန်ဖိုးတစ်ခု (value) မဟုတ်ဘဲ၊ တန်ဖိုး မရှိခြင်း (absence of a value) ဖြစ်ပါတယ်။
- NULL ပါဝင်လာရင် နှိုင်းယှဉ်မှုတွေ (logical comparisons)၊ သင်္ချာတွက်ချက်မှုတွေ (arithmetic) နဲ့ အချက်အလက် မေးမြန်းထုတ်ယူမှု ရလဒ်တွေ (query results) အပေါ် သက်ရောက်မှု ရှိပါတယ်။
- SQL language က NULL တွေကို အခြေအနေအလိုက် မတူညီတဲ့ နည်းလမ်းတွေနဲ့ ကိုင်တွယ်ပါတယ်။ (ဥပမာ- စုပေါင်းတွက်ချက်မှု (aggregate functions) တွေမှာ တစ်မျိုး၊ WHERE clause တွေမှာ တစ်မျိုး ကိုင်တွယ်တာမျိုး)။
- NULL တွေကို မှန်ကန်စွာ ကိုင်တွယ်တတ်ဖို့ဆိုတာ တိကျမှန်ကန်တဲ့ queries အတွက် မရှိမဖြစ် လိုအပ်ပါတယ်။
</Aside>
