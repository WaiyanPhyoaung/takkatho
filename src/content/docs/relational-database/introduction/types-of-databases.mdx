## Database System အမျိုးအစားများ

Database အားလုံးက အလုပ်တစ်မျိုးတည်းကိုပဲ လုပ်တာ မဟုတ်ပါဘူး။ တချို့ Database တွေက မြန်ဆန်မှုနှင့် data တွေကို အမြဲတမ်းပြင်ဆင်ပြောင်းလဲမှုတွေလုပ်နိုင်ဖို့အတွက် တည်ဆောက်ထားပြီး 
နေ့စဉ်လုပ်ငန်းဆောင်တာတွေကို လျင်မြန်စွာ အလုပ်လုပ်နိုင်အောင် ကိုင်တွယ်ဖြေရှင်းပေးပါတယ်။ တခြား Database တွေကျတော့ data တွေကို နက်ရှိုင်းစွာ ခွဲခြမ်းစိတ်ဖြာဖို့အတွက် 
ပြုလုပ်ထားပြီးတော့ historical data တွေထဲကနေ Trend တွေနှင့် Pattern တွေကို ရှာဖွေနိုင်စေပါတယ်။ Database တွေကို လက်တွေ့မှာ ဘယ်လိုအသုံးပြုလဲဆိုတဲ့အပေါ် မူတည်ပြီး 
အဓိက Database အမျိုးအစားနှစ်ခုဖြစ်တဲ့ လုပ်ငန်းဆောင်တာဆိုင်ရာ Database များ (Operational Databases - OLTP)နှင့် ခွဲခြမ်းစိတ်ဖြာမှုဆိုင်ရာ Databases များ (Analytical Databases - OLAP) ကို
လေ့လာကြည့်ရအောင်။

## 1. Operational Databases (OLTP – Online Transaction Processing)
လူစည်ကားတဲ့ ကုန်တိုက်ကြီးတစ်ခုရဲ့ ဈေးဝယ်သူကျနေတဲဲ့အချိန် အလုပ်များနေတာကို ဒါမှမဟုတ် အွန်လိုင်းအော်ဒါတွေ အဆက်မပြတ်ဝင်နေတာကို မြင်ယောင်ကြည့်လိုက်ပါ။ 
Operational Database တွေဟာ ဒစ်ဂျစ်တယ်ကမ္ဘာရဲ့ အဓိကအလုပ်လုပ်ဆောင်ပေးတဲ့သူတွေဖြစ်ပြီးတော့ နေ့စဉ်လုပ်ငန်းဆောင်တာတွေအတွက် အခြေခံအုတ်မြစ်ဖြစ်ပါတယ်။  
၎င်းတို့ဟာ ဒစ်ဂျစ်တယ်စီးပွားရေးရဲ့ အရေးပါတဲ့ အစိတ်အပိုင်းတစ်ခုဖြစ်တဲ့ ပစ္စည်းဝယ်ယူမှုတွေကို စီမံဆောင်ရွက်တာ၊ ကြိုတင်မှာယူမှုတွေ ပြုလုပ်တာမျိုး၊  ဆေးရုံတွင် လူနာမှတ်တမ်းတွေကိုသိမ်းဆည်းတာမျိုး၊ 
ဒါမှမဟုတ် ကုန်ပစ္စည်းစာရင်းတွေကို စီမံခန့်ခွဲတာမျိုးစတဲ့ အချိန်နှင့်တပြေးညီ လုပ်ငန်းဆောင်တာတွေကို စီမံခန့်ခွဲဖို့အတွက် ဒီဇိုင်းထုတ်ထားပါတယ်။

### Key Features
* **ပြောင်းလဲနေသော Data (Dynamic Data):** အမြဲတမ်း ပြောင်းလဲနေတဲ့ အချက်အလက်တွေကို သိမ်းဆည်းထားပြီး၊ လုပ်ငန်းဆောင်တာတွေရဲ့ အခြေအနေကို အချိန်နဲ့တပြေးညီ ဖော်ပြပေးပါတယ်။
* **CRUD လုပ်ဆောင်ချက်များ (CRUD Operations):** OLTP databases တွေဟာ Create (တည်ဆောက်ခြင်း)၊ Read (ဖတ်ရှုခြင်း)၊ Update (ပြင်ဆင်ခြင်း)နှင့် Delete (ဖျက်ခြင်း) ဆိုတဲ့ CRUD လုပ်ဆောင်ချက်တွေကို အမြန်ဆုံးနဲ့ စနစ်အကျဆုံး ဖြစ်အောင် အဓိကထားပြီး လုပ်ဆောင်ပေးပါတယ်။
* **လျင်မြန်ပြီး ယုံကြည်စိတ်ချရသော လုပ်ငန်းဆောင်တာများ (Fast & Reliable Transactions):**  Transaction တစ်ခုချင်းစီ (ဥပမာ- ဘဏ်ငွေလွှဲခြင်း၊ ဈေးဝယ်လှည်းထဲသို့ ပစ္စည်းထည့်ခြင်း)ကို လျင်လျင်မြန်မြန်နဲ့ မှန်မှန်ကန်ကန် စိတ်ချယုံကြည်စွာ စီမံဆောင်ရွက်ပေးနိုင်ပါတယ်။
* **အချိန်နှင့်တပြေးညီ တိကျမှု (Real-time Accuracy):** data အသစ်တွေဝင်ရောက်လာတာနဲ့ သို့မဟုတ် data အပြောင်းအလဲဖြစ်တာနဲ့ တစ်ပြိုင်နက်တည်း ချက်ချင်း database ထဲမှာ မှတ်တမ်းတင်ရတာမျိုး၊ သင်ကြည့်လိုက်တဲ့ အချိန်မှာ ရရှိတဲ့ data တိုင်းဟာ အမှန်ဆုံးနဲ့ နောက်ဆုံး(up to date) အခြေအနေကို ဖော်ပြနေရတာမျိုး 
လိုအပ်သော system များအတွက် အရေးကြီးတဲဲ့ database အမျိုးအစားဖြစ်ပါတယ်။ 

### လက်တွေ့တွင် အသုံးများသော Application များ
- **လက်လီနှင့် အီလက်ထရောနစ် ကူးသန်းရောင်းဝယ်ရေး (Retail & E-commerce):** ကုန်ပစ္စည်းစာရင်းများကို စီမံခန့်ခွဲခြင်း၊ ရောင်းအား Transaction များကို စီမံဆောင်ရွက်ခြင်း၊ အွန်လိုင်းအော်ဒါများနှင့် ပို့ဆောင်မှုများကို မှတ်တမ်းတင်ခြင်း။
- **ကျန်းမာရေးစောင့်ရှောက်မှု (Healthcare):** လူနာမှတ်တမ်းများ သိမ်းဆည်းခြင်းနှင့် ပြင်ဆင်ခြင်း၊ ချိန်းဆိုထားသည့် ဆေးခန်းပြချိန်များကို စီမံခန့်ခွဲခြင်း၊  ဆေးဘက်ဆိုင်ရာပစ္စည်းများ အရေအတွက်၊ ကုန်ဆုံးရက်၊ ဝင်ရောက်မှုနဲ့ ထွက်ခွာမှု မှတ်တမ်းတွေကို စနစ်တကျ မှတ်တမ်းတင်ခြင်း။
- **ဘဏ်လုပ်ငန်း (Banking):** ဘဏ်စာရင်းလက်ကျန်များ ကိုင်တွယ်ခြင်း၊ ငွေသွင်းခြင်း၊ ငွေထုတ်ခြင်းနှင့် လွှဲပြောင်းခြင်းများ စီမံဆောင်ရွက်ခြင်း။
- **ပို့ဆောင်ရေးနှင့် ထောက်ပံ့ရေး (Logistics & Supply Chain):** ကုန်ပစ္စည်းတင်ပို့မှုများ မှတ်တမ်းတင်ခြင်း၊ ဂိုဒေါင်သိုလှောင်မှုများကို စီမံခန့်ခွဲခြင်း၊ ပစ္စည်းပို့ဆောင်ရေးလမ်းကြောင်းများကို အကောင်းဆုံးတွက်ချက်ခြင်း။

### Technical Features
- **High Transaction Volume:** Transaction ပမာဏအမြောက်အများကို တစ်ပြိုင်နက်တည်း ကိုင်တွယ်နိုင်အောင် ဒီဇိုင်းထုတ်ထားပါတယ်။
- **Data Integrity & Concurrency:** data တွေ အမြဲတမ်း မှန်ကန်တိကျပြီး ကိုက်ညီမှုရှိနေစေဖို့ (Data Consistency) (ဥပမာ - ဘဏ်စာရင်းတစ်ခုမှာ ငွေထုတ်လိုက်ရင် စာရင်းထဲက ပိုက်ဆံပမာဏကို လျှော့ပေးရမယ်၊ 
ငွေထုတ်တဲ့ မှတ်တမ်းကိုလည်း သိမ်းဆည်းရမယ်။ အဲဒီလို data အားလုံးက အမြဲတမ်း မှန်ကန်စွာ ကိုက်ညီနေဖို့လိုအပ်ပါတယ်။) နှင့် သုံးစွဲသူအများအပြားက data တွေကို တစ်ပြိုင်နက်တည်း ပြဿနာမရှိဘဲ ဝင်ရောက်အသုံးပြုခြင်း၊ ပြောင်းလဲခြင်းတို့ လုပ်ဆောင်နိုင်အောင် ပြုလုပ်ထားပါတယ်။
- **Relational (SQL):** အများအားဖြင့် OLTP ကို Relational Database Management Systems (RDBMS) တွေကို အသုံးပြုပြီး တည်ဆောက်ထားပါတယ်။ အဲဒီ RDBMS တွေက data တွေကို စီမံခန့်ခွဲဖို့အတွက် SQL (Structured Query Language) ကို သုံးပါတယ်။

### OLTP ကို ဘယ်အချိန်မှာ အသုံးပြုမလဲ။
သင်၏ application က အချိန်နှင့်တပြေးညီ ဒေတာပြင်ဆင်မှုများနှင့် လုပ်ငန်းဆောင်တာများကို လျင်မြန်စွာ၊ ယုံကြည်စိတ်ချစွာ ဆောင်ရွက်ရန် လိုအပ်သောအခါ OLTP ကိုရွေးချယ်နိုင်ပါတယ်။

## 2. Analytical Databases (OLAP – Online Analytical Processing)

Operational Database တွေရဲ့ ဆန့်ကျင်ဘက်အနေဖြင့် Analytical Database တွေကတော့ ခွဲခြမ်းစိတ်ဖြာမှုနှင့် နက်ရှိုင်းစွာ နားလည်သဘောပေါက်မှုများအတွက် တည်ဆောက်ထားပါတယ်။ 
၎င်းတို့ကို historical Data ပမာဏများစွာကို ခွဲခြမ်းစိတ်ဖြာဖို့ ဒီဇိုင်းထုတ်ထားပြီး စီးပွားရေးလုပ်ငန်းတွေနှင့် သုတေသီများအတွက် ခေတ်ရေစီးကြောင်းများ (Trend) ရှာဖွေနိုင်ဖို့၊ Data Behavior တွေအကြောင်း နားလည်နိုင်ဖို့ ကူညီပေးပါတယ်။
<br></br>
၎င်းတို့ဟာ အောက်ပါကဲ့သို့သော ရှုပ်ထွေးတဲ့မေးခွန်းတွေကို ဖြေဆိုဖို့အတွက် အကောင်းဆုံးဖြစ်အောင် ပြုလုပ်ထားပါတယ်။
- "လွန်ခဲ့တဲ့ ဘဏ္ဍာရေးနှစ်က ကျွန်တော်တို့ရဲ့ ရောင်းအား စွမ်းဆောင်ရည် ဘယ်လိုတိုးတက်ပြောင်းလဲလာလဲ။"
- "လွန်ခဲ့တဲ့ သုံးလပတ်မှာ ဘယ်ကုန်ပစ္စည်းအမျိုးအစားတွေက Customer စိတ်ဝင်စားမှု အမြင့်ဆုံးဖြစ်ခဲ့လဲ။"
- "ကျွန်တော်တို့ရဲ့ အသုံးပြုသူအသစ်အများစုက ဘယ်ဒေသတွေကဖြစ်သလဲ။" 

### Key Features
- **မပြောင်းလဲသော/သမိုင်းကြောင်းဆိုင်ရာ ဒေတာ (Static/Historical Data):** အဓိကအားဖြင့် မှတ်တမ်းတင်ပြီးတာနှင့် အလွယ်တကူ မပြောင်းလဲတော့တဲ့ သမိုင်းဝင် ဒါမှမဟုတ် Static Data အမြောက်အများကို အဓိက သိမ်းဆည်းပါတယ်။
- **ရှုပ်ထွေးသော query များအတွက် အကောင်းဆုံးလုပ်ဆောင်နိုင်ခြင်း (Optimized for Complex Queries):** များစွာသော အရင်းအမြစ်များမှ ဒေတာများကို စုစည်းခြင်း(aggregating)၊ အကျဉ်းချုပ်ခြင်း(summarizing)နှင့် ချိတ်ဆက်ခြင်း(joining)တို့ ပါဝင်သည့် ရှုပ်ထွေးသော query များကို ထိထိရောက်ရောက် လုပ်ဆောင်နိုင်ရန် ဒီဇိုင်းထုတ်ထားပါတယ်။
- **ထိုးထွင်းသိမြင်မှုများ (Powerful Insights):** သုံးစွဲသူများအား ဒေတာကို "အမျိုးမျိုးသော ရှုထောင့်မှကြည့်ရှုနိုင်ခြင်း" (slice and dice)၊ အသေးစိတ်ကြည့်ရှုနိုင်ခြင်း (drill-downs) နှင့် စီးပွားရေးဆိုင်ရာ ထိုးထွင်းသိမြင်မှုများ (business intelligence) အတွက် အစီရင်ခံစာများ (aggregated reports) ထုတ်ပေးနိုင်ပါတယ်။

### လက်တွေ့တွင် အသုံးများသော Application များ
- **စျေးကွက်ခွဲခြမ်းစိတ်ဖြာမှု (Marketing Analytics):** ယခင် Campaign တွေရဲ့ ထိရောက်မှု၊ Customer ခွဲခြားသတ်မှတ်ခြင်းနှင့် အနာဂတ် ခေတ်ရေစီးကြောင်း(Trend)များကို ခန့်မှန်းခြင်း။
- **သိပ္ပံဆိုင်ရာ သုတေသန (Scientific Research):** Time-series Data တွေကို လေ့လာခြင်း၊ large-scale simulationsနှင့် စမ်းသပ်မှုရလဒ်တွေထဲက Pattern တွေကို ဖော်ထုတ်ခြင်း။
- **အမှုဆောင်အဆင့် ဆုံးဖြတ်ချက်ချမှတ်ခြင်း (Executive Decision-Making):**  ကုမ္ပဏီစွမ်းဆောင်ရည်၊ စျေးကွက်ခွဲခြမ်းစိတ်ဖြာမှုနှင့် မဟာဗျူဟာမြောက် စီမံကိန်းရေးဆွဲခြင်းအတွက် Dashboard တွေနှင့် Report တွေ ပံ့ပိုးပေးခြင်း။
- **စီးပွားရေး ထိုးထွင်းသိမြင်မှု (Business Intelligence - BI):** အဖွဲ့အစည်းဆိုင်ရာ အဆင့်အားလုံးတွင် ဒေတာကိုအခြေခံတဲ့ ဆုံးဖြတ်ချက်ချမှတ်ခြင်းအတွက် ထိုးထွင်းသိမြင်မှု(insights) တွေကို ပံ့ပိုးပေးတဲ့ Tools တွေကို လုပ်ဆောင်ရည်တိုးမြှင့်ပေးခြင်း။

## Relational Database

### Relational Database ဆိုတာ ဘာလဲ။

Relational Database Model ဆိုတာက ဒီနေ့ခေတ် Database Management မှာ အသုံးအများဆုံး Database Model ဖြစ်ပါတယ်။
ဘဏ္ဍာရေး(Finance)၊ ကျန်းမာရေးစောင့်ရှောက်မှု(Healthcare)၊ ပညာရေး(Education) ဒါမှမဟုတ် လက်လီရောင်းဝယ်ရေး(Retail) အစရှိသည့် နယ်ပယ်တွေမှာ
relational database ကိုအသုံးပြုပြီး ဒေတာတွေကို စီမံခန့်ခွဲနေကြပါတယ်။

> အဓိကအချက်ကတော့ ဒေတာတွေကို Relations အဖြစ် စုစည်းထားတာဖြစ်ပြီး၊ အဲဒီ Relations တွေကို Table အဖြစ်လည်း စဉ်းစားနိုင်ပါတယ်။ ဒီ Table တွေကို အောက်ပါအရာတွေနဲ့ ဖွဲ့စည်းထားပါတယ်။
> - Rows (Records သို့မဟုတ် tuples လို့လည်း ခေါ်ပါတယ်): အရာဝတ္ထုတစ်ခု၊ အချက်အလက်တစ်ခု ဒါမှမဟုတ် Entity တစ်ခုကို ကိုယ်စားပြုပါတယ်။ ဥပမာ - လူတစ်ယောက်၊ ပစ္စည်းတစ်ခု။
> - Columns (attributes သို့မဟုတ် fields လို့လည်း ခေါ်ပါတယ်): ဒေတာရဲ့ အမျိုးအစား ဒါမှမဟုတ် characteristic တစ်ခုကို ကိုယ်စားပြုပါတယ်။ ဥပမာ - နာမည်၊ အသက်၊ အရောင်။
>
> Table (ဒါမှမဟုတ် Relation) တိုင်းမှာ Primary Key တစ်ခုစီရှိပါတယ်။
> Primary key က table မှာရှိတဲ့ row တစ်ခုချင်းစီကို တိတိကျကျ ဆွဲထုတ်ပေးနိုင်တဲ့ cloumn တစ်ခု ဒါမှမဟုတ် column အစုအဝေးတစ်ခု ဖြစ်ပါတယ်။

### Table တစ်ခုနဲ့တစ်ခု ဘယ်လို ချိတ်ဆက်ထားသလဲ။ 

Relational Database တွေရဲ့ အဓိကစွမ်းဆောင်ရည်ကတော့ သီးခြားစီရှိနေတဲ့ table တွေကို relationships တွေကနေတစ်ဆင့် ဘယ်လို ချိတ်ဆက်နိုင်လဲဆိုတဲ့ အပေါ်မှာ မူတည်ပါတယ်။
Relationship တွေကို များသောအားဖြင့် table နှစ်ခုမှာရှိတဲ့ တူညီတဲ့ fields (matching values) တွေကနေ တည်ဆောက်တာဖြစ်ပါတယ်။ (ဥပမာ- Primary Key နှင့် Foreign Key)

**ပိုပြီးနားလည်အောင် ဥပမာနဲ့ ပြောရရင်။**

စာသင်ကျောင်းတစ်ခုရဲ့ အချက်အလက်တွေကို မှတ်ထားတယ်ဆိုပါစို့။

ကျောင်းသားစာရင်း (Student Table) ဆိုတဲ့ table တစ်ခုမှာ ကျောင်းသားတစ်ယောက်ချင်းစီတိုင်းအတွက် ကျောင်းသားကုတ်နံပါတ် (Primary Key) တွေရှိမယ်။ (ဥပမာ - S001, S002 စသည်ဖြင့်)
သင်တန်းစာရင်း (Course Enrollment Table) ဆိုတဲ့ နောက်ထပ် table တစ်ခုရှိမယ်။ ပြီးတော့ ဘယ်ကျောင်းသားက ဘယ်သင်တန်းတက်တယ်ဆိုတာ မှတ်ထားချင်တယ်ဆိုရင် အဲဒီ သင်တန်းစာရင်း table မှာ ကျောင်းသားတွေရဲ့ 
ကျောင်းသားကုတ်နံပါတ်ကို Foreign Key အနေနဲ့ ထည့်ပေးမယ်။ ဒီ foreign key က ကျောင်းသားစာရင်းထဲက primary key နဲ့ သွားတူနေတာပေါ့။
ဒီလိုချိတ်ဆက်ထားခြင်းဖြင့် ဘယ်ကျောင်းသားက ဘယ်သင်တန်းတက်တယ်ဆိုတာကို table နှစ်ခုကို ဆက်စပ်ပြီး အလွယ်တကူ သိနိုင်ပါတယ်။ 
ဒါက Relational Databases ရဲ့ အားသာချက်ပဲ ဖြစ်ပါတယ်။

အသုံးများသော Relationship အမျိုးအစား သုံးခုရှိပါတယ်။

- **One-to-One (1:1)** <br />
Table A ထဲက Record တစ်ခုချင်းစီက Table B ထဲက Record တစ်ခုတည်းနဲ့သာ ဆက်စပ်နေတယ်။  <br />
ဥပမာ- User တစ်ယောက်နဲ့ သူ့ရဲ့ တစ်ခုတည်းရှိသော Profile။ <br />
- **One-to-Many (1:N)** <br />
Table A ထဲက Record တစ်ခုက Table B ထဲက Record များစွာနဲ့ ဆက်စပ်နေတယ်။  <br />
ဥပမာ- Customer တစ်ယောက်နဲ့ သူ့ရဲ့ Order များ။ <br />
- **Many-to-Many (M:N)** <br />
Table A ထဲက Record တွေက Table B ထဲက Record များစွာနဲ့  ဆက်စပ်နေနိုင်သလို၊ Table B ထဲက Record တွေကလည်း Table A ထဲက Record များစွာနဲ့ ဆက်စပ်နေနိုင်တယ်။ <br />
ဥပမာ- ကျောင်းသားတွေနဲ့ သူတို့တက်ရောက်နေတဲ့ Course တွေ။ <br />

ဒီ relationships ကနေတစ်ဆင့် ချိတ်ဆက်ထားတဲ့ table ပေါင်းများစွာထဲက အချက်အလက်တွေကို လိုအပ်သလို ရှာဖွေကြည့်ရှုနိုင်ပါတယ်။ တကယ်လို့ အဲဒီ table တွေက တိုက်ရိုက်ချိတ်ဆက်မထားရင်တောင်မှ
shared keys ကနေတစ်ဆင့် ဆက်သွယ်မှု (logical connection) ရှိနေသမျှကာလပတ်လုံး ရှာဖွေကြည့်ရှုနိုင်ပါတယ်။


## Relational Database Management System

Relational Database Management System (RDBMS) ဆိုသည်မှာ relational database တစ်ခုကို ဖန်တီးရန်၊ စီမံခန့်ခွဲရန်၊ data တွေ ပြင်ဆင်ရန်နှင့် အပြန်အလှန် အလုပ်လုပ်ရန် ကူညီပေးသည့် ဆော့ဖ်ဝဲလ်(software) တစ်ခုဖြစ်ပါသည်။ 
Relational Model သည် data တွေကို သဘောတရားအရ ဘယ်လို စီစဉ်ထားလဲ (Table များ၊ Row များ၊ Relationship များ) ဆိုတာကို ဖော်ပြပေးပေမယ့်၊ RDBMS က အဲဒီအရာအားလုံးကို လက်တွေ့မှာ အလုပ်လုပ်နိုင်အောင် လုပ်ဆောင်ပေးတဲ့ tool ဖြစ်ပါတယ်။
RDBMS ကို database ရဲ့ ဦးနှောက်လို့ တွေးကြည့်လို့ရပါတယ်။ Data သိမ်းဆည်းတာ၊ ပြန်လည်ထုတ်ယူတာ၊ လုံခြုံရေး၊ တိကျမှန်ကန်မှု စတာတွေကို ကိုင်တွယ်ဆောင်ရွက်ပေးပါတယ်။
ဒါကြောင့် သုံးစွဲသူ(users) တွေနဲ့ ဆော့ဖ်ဝဲလ် (applications) တွေအနေနဲ့ data တွေကို ဘာလုပ်ချင်လဲဆိုတဲ့အပေါ်မှာပဲ အာရုံစိုက်နိုင်ပြီး၊ နောက်ကွယ်မှာ data တွေ ဘယ်လိုသိမ်းဆည်းထားလဲဆိုတာကို ခေါင်းစားနေစရာ မလိုတော့ပါဘူး။

>Relational Database Model ကတော့ ကျွန်တော်တို့ကို သဘောတရားပိုင်းဆိုင်ရာ (theory) ကို ပေးပါတယ်။ ဥပမာ - data တွေကို table တွေထဲ သိမ်းတာ၊ keys တွေကနေတစ်ဆင့် relationship တွေ ပြုလုပ်တာ၊ data တွေ မှန်ကန်တိကျဖို့ စည်းမျဉ်းစည်းကမ်းတွေ သတ်မှတ်တာ စတာတွေပေါ့။
>အဲဒီသဘောတရားတွေကို လက်တွေ့မှာ အလုပ်ဖြစ်အောင် လုပ်ပေးတာကတော့ RDBMS ဆိုတဲ့ ကိရိယာပါပဲ။ RDBMS က relational database တွေကို ထိထိရောက်ရောက်နဲ့ လုံလုံခြုံခြုံ တည်ဆောက်ဖို့၊ စီမံခန့်ခွဲဖို့နဲ့ အလုပ်လုပ်ဖို့ ကူညီပေးပါတယ်။

🔎 ဒီလိုတွေးကြည့်နိုင်ပါတယ်။
relational database ဟာ သင့်အိမ်ရဲ့ အဆောက်အအုံပုံစံ (structure) ဖြစ်တယ်ဆိုရင်၊ RDBMS ဆိုတာက အဲဒီအိမ်ကို ဆောက်လုပ်ဖို့နဲ့ ပြုပြင်ထိန်းသိမ်းဖို့အတွက် လိုအပ်တဲ့ ကိရိယာအစုံအလင် (toolbox)၊ လုပ်သားအဖွဲ့ (crew) နဲ့ စမတ်အိမ်စနစ် (smart home system) တွေပဲ ဖြစ်ပါတယ်။
