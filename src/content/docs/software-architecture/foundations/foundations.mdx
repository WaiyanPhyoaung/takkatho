### Architecture ကို အဓိပ္ပာယ်ဖွင့်ဆိုခြင်း - အလုံးစုံခြုံငုံကြည့်မြင်ခြင်း
အနှစ်ချုပ်အားဖြင့် Software Architecture ဆိုသည်မှာ software system တစ်ခု၏ high-level တည်ဆောက်ပုံများနှင့် ၎င်းတို့ကိုဖန်တီးရာတွင် ချမှတ်ခဲ့သော အရေးကြီးသည့် ဆုံးဖြတ်ချက်များကို ရည်ညွှန်းပါသည်။ code အသေးစိတ်အချက်အလက်များအကြောင်းမဟုတ်ဘဲ၊ အရာအားလုံးကို စုစည်းထိန်းသိမ်းထားသည့် အလုံးစုံ (skeleton) အကြောင်းဖြစ်သည်။

#### ဥပမာနှိုင်းယှဉ်ချက် - အိမ်တစ်လုံးဆောက်ခြင်း

အိမ်တစ်လုံးဆောက်တာကို စဉ်းစားကြည့်ပါ။ Software architecture ဆိုတာ အဲဒီအိမ်ရဲ့ ပုံစံကြမ်း (blueprint) ပါပဲ။ ၎င်းက အောက်ပါတို့ကို သတ်မှတ်ပေးပါတယ်:

- အိမ်ရဲ့ အုတ်မြစ်။
- အထပ် ဘယ်လောက်ရှိမလဲ။
- အဓိကအခန်းတွေ (မီးဖိုချောင်၊ အိပ်ခန်း) ဘယ်နေရာမှာရှိမလဲ။
- မီးလိုင်း၊ ရေလိုင်း၊ လေဝင်လေထွက်စနစ်တွေ ဘယ်လိုချိတ်ဆက်ပြီး တည်ဆောက်ပုံတစ်လျှောက် သွယ်တန်းသွားမလဲ။
- ပုံစံကြမ်းက နံရံအတွက် ဘယ်ဆေးအမျိုးအစားသုံးမလဲ၊ မီးဖိုချောင်မှာ ဘယ်လိုဘုံဘိုင်ခေါင်းတပ်မလဲဆိုတာတွေကို မသတ်မှတ်ပါဘူး—ဒါတွေက အသေးစိတ်အဆင့် (lower-level details) တွေပါ။ **Architecture ဆိုတာက နောက်ပိုင်းမှာ ပြောင်းလဲဖို့ ခက်ခဲပြီး ကုန်ကျစရိတ်များတဲ့** အခြေခံတည်ဆောက်ပုံအကြောင်းဖြစ်ပါတယ်။ အခန်းတစ်ခန်းကို ဆေးအလွယ်တကူပြန်သုတ်လို့ရပေမယ့်၊ အိမ်တစ်လုံးကို အထပ်တစ်ထပ်ထပ်တိုးဖို့ကတော့ အလွန်ကြီးမားတဲ့ လုပ်ငန်းစဉ်တစ်ခုဖြစ်ပါတယ်။

### Architecture, Design, နှင့် Code တို့ကို ခွဲခြားနားလည်ခြင်း
ဤအသုံးအနှုန်းများကို မကြာခဏ ရောထွေးသုံးစွဲလေ့ရှိသော်လည်း၊ ၎င်းတို့သည် မတူညီသော အသေးစိတ်အဆင့်များကို ကိုယ်စားပြုသည်။

- **Architecture :** အဓိကဇုန်များ (လူနေ၊ စီးပွားရေး)၊ အဓိကအဝေးပြေးလမ်းမကြီးများ၊ နှင့် ဓာတ်အားပေးစက်ရုံများ၊ ရေပေးဝေရေးစခန်းများ၏ တည်နေရာများကို သတ်မှတ်သည်။ ဤဆုံးဖြတ်ချက်များက နောက်ပိုင်းကျမှ ပြောင်းလဲရန် အလွန်ခက်ခဲသည်။

    - ဥပမာ: "ကျွန်ုပ်တို့၏ application သည် event များဖြင့် ဆက်သွယ်သော **microservices architecture** ဖြစ်ရမည်။"

- **Design :** မြို့ပြစီမံကိန်းအတွင်းရှိ အဆောက်အဦးတစ်ခုတည်းအတွက် ကြမ်းခင်းပုံစံကို သတ်မှတ်သည်။ ၎င်းသည် မြို့ပြစီမံကိန်း၏ စည်းမျဉ်းများကို လိုက်နာသော်လည်း၊ component တစ်ခု၏ တည်ဆောက်ပုံကိုသာ ကိုင်တွယ်သည်။

    - ဥပမာ: "**User Service** သည် API, business logic layer, နှင့် data access layer တို့ပါဝင်သော layered pattern ကို အသုံးပြုမည်။"

- **Code :** အိမ်ဆောက်ရန် အသုံးပြုသော တိကျသည့်ပစ္စည်းများနှင့် implementation အသေးစိတ်များကို သတ်မှတ်သည်။

    - ဥပမာ: **`class UserService { ... }`**

### Software Architect တစ်ဦး၏ အခန်းကဏ္ဍနှင့် တာဝန်များ
Architect တစ်ဦး၏ အခန်းကဏ္ဍသည် ပုံဆွဲရုံသက်သက်ထက် ပိုပါသည်။ ၎င်းသည် အောက်ပါ အဓိကတာဝန်များပါဝင်သော ဦးဆောင်မှုအခန်းကဏ္ဍတစ်ခု ဖြစ်သည်:

- **အဓိကကျသော နည်းပညာဆိုင်ရာ ဆုံးဖြတ်ချက်များ ချမှတ်ခြင်း -** Project ၏ အနာဂတ်ကို ပုံဖော်မည့် အရေးကြီးသော **architectural ရွေးချယ်မှုများ**ကို ဖော်ထုတ်၍ ဆုံးဖြတ်ချက်ချမှတ်ခြင်း။

- **အပေးအယူမျှတမှုများကို သုံးသပ်ရွေးချယ်ခြင်း -** Architecture များရဲ့ အားနည်းချက် အားသာချက်များကို နားလည်ပြီး၊ business နှင့် technical ရည်မှန်းချက်များကို ချိန်ညှိဆောင်ရွက်ခြင်း။

- **ရည်မှန်းချက် (Vision) ကို ရှင်းလင်းစွာ ဆက်သွယ်ပြောဆိုခြင်း -** Team ရှိ developer တိုင်းက architecture ကို နားလည်စေခြင်း၊ နောက်ကွယ်မှ အကြောင်းပြချက်များကို သိရှိစေခြင်း နှင့် သူတို့၏အလုပ်က မည်သို့ အကျိုးသက်ရောက်မှုရှိခြင်းနှင့် အံဝင်ခွင်ကျဖြစ်သည်ကို သေချာစေခြင်း။

- **နည်းပညာပိုင်းဆိုင်ရာ ဦးဆောင်မှုပေးခြင်းနှင့် လမ်းညွှန်ပေးခြင်း (Mentorship) -** Team ကို လမ်းညွှန်ခြင်း၊ နည်းပညာစံနှုန်းများ သတ်မှတ်ပေးခြင်း၊ နှင့် developer များ၏ ကျွမ်းကျင်မှုများ တိုးတက်လာအောင် ကူညီပေးခြင်း။
