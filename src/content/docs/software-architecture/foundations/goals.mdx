  Quality attributes (non-functional requirements) များနှင့် အားသာချက် အားနည်းချက် (trade-offs) များကို နားလည်ပြီး Architectural ဆုံးဖြတ်ချက်များကို ချမှတ်တတ်စေရန်။

### Feature များအလွန် - System တစ်ခု "ဘယ်လောက်ကောင်းကောင်း" အလုပ်လုပ်သလဲ သတ်မှတ်ခြင်း
**Architecture** ဆိုသည်မှာ system တစ်ခုက ဘာတွေလုပ်ဆောင်ပေးနိုင်သလဲ (features) ဆိုတာထက်၊ ၎င်းအလုပ်များကို **ဘယ်လောက်ကောင်းကောင်း လုပ်ဆောင်နိုင်သလဲ** ဆိုတာကို အဓိကထားပါတယ်။ ဤ "ဘယ်လောက်ကောင်းကောင်း" ဆိုတဲ့ လက္ခဏာရပ်များကို **Quality Attributes သို့မဟုတ် "-ilities"** ဟုခေါ်သည်။ ၎င်းတို့သည် သင်၏ architectural ပုံစံကြမ်းကို ပုံဖော်ပေးသည့် **အဓိကရည်မှန်းချက်များ** ဖြစ်ကြသည်။

အရေးအကြီးဆုံး quality attributes အချို့မှာ-

- **Performance (စနစ်၏တုံ့ပြန်မှုအမြန်နှုန်း) ⏱️**

    - **ဘာကိုဆိုလိုသလဲ -** System က request တစ်ခုကို ဘယ်လောက်မြန်မြန်တုံ့ပြန်သလဲ။ ၎င်းကို **latency** (ကြန့်ကြာမှု) နှင့် **throughput** (အချိန်တစ်ခုအတွင်း ပြီးမြောက်သောအလုပ်) တို့ဖြင့် တိုင်းတာသည်။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "သုံးစွဲသူတစ်သန်း တစ်ပြိုင်နက်တည်းရှိနေရင်တောင်၊ user ရဲ့ search result တွေက ဘယ်လောက်မြန်မြန်ပေါ်လာရမလဲ?"

- **Scalability (တိုးချဲ့နိုင်ခြင်း) 📈**

    - **ဘာကိုဆိုလိုသလဲ -** System က တိုးပွားလာသော သုံးစွဲသူများနှင့် data များ ကို စွမ်းဆောင်ရည် လုံးဝမကျဆင်းဘဲ ထိထိရောက်ရောက် ကိုင်တွယ်နိုင်စွမ်း ရှိ၊ မရှိ။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "ကျွန်ုပ်တို့၏ သုံးစွဲသူအရေအတွက်က တစ်ညအတွင်း နှစ်ဆတိုးသွားခဲ့ရင်၊ ကျွန်ုပ်တို့ system က မကျဘဲ သို့မဟုတ် မနှေးကွေးဘဲ ကိုင်တွယ်နိုင်ပါ့မလား?"

- **Availability / Reliability (အမြဲတမ်းရရှိနိုင်မှု / ယုံကြည်စိတ်ချရမှု) ✅**

    - **ဘာကိုဆိုလိုသလဲ -** System က ကျသွားခြင်းမရှိဘဲ မှန်မှန်ကန်ကန်လည်ပတ်နေသော အချိန်ရာခိုင်နှုန်း။ များသောအားဖြင့် **"nines"** (ဥပမာ - 99.9% uptime) ဖြင့် တိုင်းတာသည်။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "Server တစ်လုံး ပျက်သွားခဲ့ရင်၊ website တစ်ခုလုံး ရပ်တန့်သွားမှာလား၊ ဒါမှမဟုတ် ဆက်လက်လည်ပတ်နိုင်မှာလား?"

- **Maintainability (ပြင်ဆင်ထိန်းသိမ်းရလွယ်ကူမှု) 🔧**

    - **ဘာကိုဆိုလိုသလဲ -** Developer များအတွက် bug များကို ရှာဖွေပြင်ဆင်ရန်၊ update များပြုလုပ်ရန်၊ နှင့် feature အသစ်များထည့်ရန် မည်မျှလွယ်ကူသနည်း။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "Developer အသစ်တစ်ယောက်က ဒီ system အပိုင်းကို နားလည်ပြီး အမှားမဖြစ်ဘဲ ပြောင်းလဲမှုတစ်ခုပြုလုပ်ဖို့ ဘယ်လောက်ကြာမလဲ?"

- **Security (လုံခြုံရေး) 🛡️**

    - **ဘာကိုဆိုလိုသလဲ -** System ၏ data နှင့် လုပ်ဆောင်ချက်များကို ခြိမ်းခြောက်မှုများနှင့် ခွင့်ပြုချက်မရှိဘဲ ဝင်ရောက်ခြင်းမှ ကာကွယ်နိုင်စွမ်း။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "ကျွန်ုပ်တို့က user password တွေကို ဘယ်လိုကာကွယ်ထားသလဲ၊ SQL injection လိုမျိုး အဖြစ်များတဲ့ တိုက်ခိုက်မှုတွေကို ဘယ်လိုကာကွယ်မလဲ?"

- **Testability (စမ်းသပ်စစ်ဆေးရလွယ်ကူမှု) 🧪**

    - **ဘာကိုဆိုလိုသလဲ -** System က မှန်မှန်ကန်ကန်အလုပ်လုပ်ကြောင်း သေချာစေရန် စမ်းသပ်စစ်ဆေးရန် မည်မျှလွယ်ကူသနည်း။
    - **Architect တစ်ဦး၏ မေးခွန်း -** "ကျွန်ုပ်တို့ system ရဲ့ အစိတ်အပိုင်းတစ်ခုချင်းစီကို သီးခြားခွဲပြီး စမ်းသပ်နိုင်လား၊ ဒါမှမဟုတ် တစ်ခုလုံးကို တစ်ခါတည်း စမ်းသပ်ရမှာလား?"

### အဓိကနိယာမ 
- Architecture ဆိုသည်မှာ Trade-offs များအကြောင်း ဖြစ်သည်။
သင်သည် တစ်ပြိုင်နက်တည်း အမြန်ဆုံး၊ အတိုးချဲ့နိုင်ဆုံး၊ အလုံခြုံဆုံး၊ နှင့် တည်ဆောက်ထိန်းသိမ်းရန် အကုန်အကျအသက်သာဆုံးဖြစ်သော system တစ်ခုကို ဘယ်တော့မှ မတည်ဆောက်နိုင်ပါ။ Architectural ဆုံးဖြတ်ချက်တိုင်းသည် အပေးအယူမျှတမှု (trade-off) တစ်ခုဖြစ်သည်။ Quality attribute တစ်ခုကို တိုးတက်အောင်လုပ်ခြင်းသည် များသောအားဖြင့် အခြားတစ်ခုကို အလျှော့ပေးရမြဲ ဖြစ်သည်။

- ဥပမာ ၁: **Performance နှင့် Security**

 System တစ်ခုကို ပိုမိုလုံခြုံအောင် လုပ်ရန်၊ သင်သည် data encryption အလွှာများစွာနှင့် ရှုပ်ထွေးသော validation check များ ထပ်ထည့်ရနိုင်သည်။

> ဤအပိုအဆင့်များက processing အချိန်ယူသောကြောင့်၊ system ကို အနည်းငယ် performance ကျ သွားစေနိုင်သည်။

- ဥပမာ ၂: **Maintainability နှင့် Performance**

System တစ်ခုကို ထိန်းသိမ်းရအလွန်လွယ်ကူအောင် လုပ်ရန်၊ သင်သည် code ကို သေးငယ်ပြီး ကောင်းမွန်စွာသတ်မှတ်ထားသော layer များနှင့် service များစွာအဖြစ် ခွဲထုတ်နိုင်သည်။
> ဤ service များက network မှတစ်ဆင့် ဆက်သွယ်ခြင်းသည် latency (ကြန့်ကြာမှု) ကိုဖြစ်ပေါ်စေပြီး၊ system ကို တသားတည်းပေါင်းစပ်ထားသော monolith တစ်ခုထက် performance ပိုနည်း သွားစေနိုင်သည်။

<Aside title="မှတ်ချက်" type="tip">
 Architect တစ်ဦး၏ အလုပ်မှာ "ပြီးပြည့်စုံသော" system တစ်ခုတည်ဆောက်ရန်သာ မဟုတ်ဘဲ၊ business ၏ အရေးအကြီးဆုံးရည်မှန်းချက်များကို နားလည်ပြီး ၎င်းတို့ကိုအောင်မြင်ရန် မှန်ကန်သော အပေးအယူမျှတမှုများ ပြုလုပ်ရန်ဖြစ်သည်။
</Aside>

<Aside title="အနှစ်ချုပ်">
- Software Architecture ဆိုသည်မှာ system တစ်ခုနှင့်ပတ်သက်၍ နောက်ပိုင်းတွင် ပြောင်းလဲရန်ခက်ခဲသော အဆင့်မြင့်ပြီး အခြေခံကျသည့် ဆုံးဖြတ်ချက်များ အစုအဝေးဖြစ်သည်။
- Architecture ၏ အဓိကရည်မှန်းချက်မှာ feature များကို အကောင်အထည်ဖော်ရန်သာမကဘဲ၊ Quality Attributes (performance, scalability, security ကဲ့သို့) များကို ဖြည့်ဆည်းပေးရန်ဖြစ်သည်။
- အရေးပါသော architectural ဆုံးဖြတ်ချက်တိုင်းတွင် project ၏ ရည်မှန်းချက်များကို အကောင်းဆုံးပြည့်မှီရန် အပြိုင်ဖြစ်နေသော quality attribute များကြား အပေးအယူမျှတမှုများ ပြုလုပ်ခြင်း ပါဝင်သည်။
</Aside>
