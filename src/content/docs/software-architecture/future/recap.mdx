<Aside title="ရည်ရွယ်ချက်">
  ဒီ Course တစ်ခုလုံးမှာ သင်ခဲ့ရတဲ့ အဓိကအချက်တွေကို ပြန်နွှေးပြီး၊ ခင်ဗျားတို့ရဲ့ Software Architecture ခရီးစဉ်ကို ဆက်လျှောက်ဖို့ ခွန်အားပေးရန် ဖြစ်ပါတယ်။
</Aside>

#### အားလုံးကို ပြန်ပေါင်းကြည့်ခြင်း - အဓိက နိယာမများ (Core Principles)
ဒီ သင်ခန်းစာကြီးတစ်ခုလုံးကနေ တခြားဟာတွေ မမှတ်မိရင်တောင် အောက်ပါ အချက်လေးတွေကိုတော့ အမြဲ မှတ်ထားစေချင်ပါတယ်:

- **Architecture ဆိုတာ ရှုပ်ထွေးမှုတွေကို စီမံခန့်ခွဲတာပါ:** ခင်ဗျားရဲ့ အဓိကအလုပ်က System ကြီးကို ဖြစ်နိုင်သမျှ အချိန်အကြာဆုံး ရိုးရှင်းနေအောင် ထိန်းသိမ်းဖို့ပါပဲ။
- **အရာအားလုံးက အပေးအယူ (Trade-off) ပါပဲ:** "အကောင်းဆုံး ပြီးပြည့်စုံတဲ့" ဖြေရှင်းချက်ဆိုတာ မရှိပါဘူး။ ကိုယ့်အခြေအနေနဲ့ အကိုက်ညီဆုံး ဖြစ်မယ့် ဖြေရှင်းချက်တွေပဲ ရှိပါတယ်။
- **နောက်ခံအခြေအနေ (Context) က အဓိကပါ:** မှန်ကန်တဲ့ Architecture ဆိုတာ ကိုယ့်လုပ်ငန်းရဲ့ ရည်မှန်းချက်၊ ဘတ်ဂျက်၊ အချိန် နဲ့ Team ရဲ့ အရည်အချင်း အပေါ်မှာပဲ မူတည်ပါတယ်။
- **"ဘာလဲ" ထက် "ဘာကြောင့်လဲ" ကို မှတ်တမ်းတင်ပါ:** အရေးကြီးတဲ့ ဆုံးဖြတ်ချက်တွေချတိုင်း ADR လို မှတ်တမ်းမျိုး ရေးခဲ့ပါ။
- **Architecture ဆိုတာ ဘယ်တော့မှ ပြီးဆုံးမသွားပါဘူး:** နည်းပညာတွေက အမြဲပြောင်းလဲနေလို့ ကိုယ့် System ကိုလည်း အမြဲ ပြန်လည်အကဲဖြတ်ပြီး တိုးတက်အောင် လုပ်နေရမှာပါ။

#### ခင်ဗျားတို့ ရှေ့ဆက်ရမယ့် ခရီး
ဒီ Course ကနေ ခင်ဗျားတို့ကို မြေပုံနဲ့ သံလိုက်အိမ်မြှောင် ပေးလိုက်ပါပြီ။ ခရီးကိုတော့ ကိုယ်တိုင် ဆက်လျှောက်ရမှာပါ။

- **ပိုပြီး နက်နက်နဲနဲ လေ့လာပါ:** ဒီထဲကနေ ကိုယ်စိတ်ဝင်စားတဲ့ Pattern တစ်ခုခု (ဥပမာ - Microservices ဒါမှမဟုတ် Event Sourcing) ကို ရွေးပြီး Personal Project သေးသေးလေး တစ်ခု ရေးကြည့်ပါ။
- **အသေးလေးကနေ စတင်ပါ:** System ကြီးတစ်ခုလုံးကို အစကနေ ပြန်ရေးဖို့ မလိုပါဘူး။ နောက်ထပ် Feature အသစ်တစ်ခု ရေးရင်ပဲဖြစ်ဖြစ် Architecture ပုံစံလေး သွင်းပြီး စဉ်းစားကြည့်ပါ။
- **ကိုယ့် Team ရဲ့ Architect ဖြစ်အောင် လုပ်ပါ:** တစ်ခုခုဆို "ဒါ ဘာကြောင့် ဒီလိုရေးရတာလဲ" လို့ စမေးပါ။ ကိုယ့်အကြံဉာဏ်တွေကို သူများနားလည်အောင် ပုံကြမ်းလေးတွေ ဆွဲပြပါ။ Trade-off တွေနဲ့ ပတ်သက်ပြီး Team ထဲမှာ ဆွေးနွေးမှုတွေ စတင်လုပ်ဆောင်ပါ။

<Aside title="အနှစ်ချုပ်">
- အနာဂတ် Architecture Trend တွေထဲမှာ **Serverless**, **Service Mesh**, နဲ့ **Data Mesh** တို့က အဓိက နေရာယူလာပါလိမ့်မယ်။
- Architect တစ်ယောက်ရဲ့ တာဝန်ဟာ **နည်းပညာကျွမ်းကျင်မှု၊ ဆက်သွယ်ပြောဆိုမှု၊ ခေါင်းဆောင်မှု၊ နဲ့ စီးပွားရေး ဗျူဟာ** ဆိုတဲ့ အချက် (၄) ချက် ပေါင်းစပ်ထားတာ ဖြစ်ပါတယ်။
- အရေးအကြီးဆုံးကတော့ ရှုပ်ထွေးမှုတွေကို ထိန်းချုပ်တာ၊ အားသာချက်/အားနည်းချက်တွေကို ချိန်ထိုးတာ နဲ့ System ကို အမြဲတမ်း တိုးတက်အောင် လုပ်တာတွေပါပဲ။ Architect တစ်ယောက်ရဲ့ ခရီးစဉ်ဆိုတာ ဒီအချက်တွေကို တစ်သက်လုံး သင်ယူပြီး လက်တွေ့အသုံးချသွားရမယ့် ခရီးစဉ်ကြီးတစ်ခု ဖြစ်ပါတယ်။
</Aside>
